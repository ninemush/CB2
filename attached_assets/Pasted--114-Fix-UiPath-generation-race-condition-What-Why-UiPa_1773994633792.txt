#114 - Fix UiPath generation race condition
What & Why
UiPath package generation never runs due to a state management race condition. When the server sends triggerUiPathGen: true followed by { done: true }, the client sets pendingUiPathGenRef.current = true and then tries to call generateUiPath(true) after a 100ms timeout. But generateUiPath() has a guard — if (isGeneratingDoc || isStreaming) return — and at the time the timeout fires, isGeneratingDoc is still true because the done handler's state cleanup hasn't flushed through React's batching yet. The trigger is silently lost. The user sees "Getting things ready..." (a generic Maintenance stage thinking message) forever, and the server never receives the POST /api/ideas/:id/generate-uipath request.

Done looks like
Sending "generate uipath package" in chat reliably triggers package generation every time — visible in server logs as POST /api/ideas/:id/generate-uipath
The progress indicator shows "Generating UiPath package..." (intent-specific) rather than "Getting things ready..." (generic stage fallback)
If the trigger arrives while streaming/generating state is still active, generation starts automatically once those states clear — no trigger is lost
Multiple rapid triggers do not start duplicate generation requests
Console logs show clear instrumentation: trigger received, trigger deferred (if blocking state active), trigger executed, generation request started
No permanent stuck states — if generation never starts after a reasonable period, the UI recovers
Out of scope
Changes to the UiPath build pipeline itself (XAML generation, quality gate, packaging)
Changes to chat LLM intent classification on the server
Other document generation flows (PDD, SDD, DHG)
Server-side changes to the chat route or generate-uipath endpoint
Tasks
Replace the fragile setTimeout-based trigger with a durable React effect that watches pendingUiPathGenRef (or a new state like pendingUiPathGen) against isStreaming and isGeneratingDoc. When the trigger is pending and both blocking states are false, fire generateUiPath(true). Clear the pending flag only after the fetch to /api/ideas/:id/generate-uipath has actually started (not before).
Fix the done handler to not clear classifiedIntent and liveStatus when a UiPath generation is pending — so the progress indicator keeps showing "Generating UiPath package..." through the handoff instead of falling back to the stage thinking message.
Make generateUiPath() distinguish three states: (a) generation already running (bail), (b) trigger pending but blocked (keep pending), (c) trigger ready to execute (consume and run). Ensure idempotency — only one generation request can be in flight at a time.
Add console.log instrumentation around the trigger lifecycle: when triggerUiPathGen is received, when the trigger is deferred because blocking state is active, when the trigger fires, and when the actual fetch starts.
Add a safety timeout (30 seconds) that clears stuck isGeneratingDoc/isStreaming state if no progress events arrive, preventing permanent UI lock if something unexpected happens.
