#254 - Dynamic Activity Schema Refresh
What & Why
Activity schemas in activity-definitions.ts are static — curated once and never updated when UiPath ships new package versions that add/change/remove properties. Version numbers update automatically via the metadata refresh cycle, but the actual property schemas (names, types, required flags, enums) stay frozen. This means the catalog gradually drifts from reality as UiPath evolves. We need a mechanism to keep schemas current without manual curation.

The core challenge: UiPath activity schemas live inside compiled .NET DLLs within NuGet packages. Extracting them requires .NET reflection, which isn't available in our Node.js runtime. This task is specifically about designing and implementing a solution to this constraint.

Done looks like
A working mechanism that can extract activity class names, properties, types, required flags, and enum values from UiPath NuGet packages
Integration with the existing 24-hour metadata refresh cycle — when a package version changes, schemas are re-extracted for that package
Extracted schemas are validated against the existing registry definitions before being accepted (sanity check: activity count shouldn't drop dramatically, known critical properties shouldn't disappear)
Fallback to existing registry definitions if extraction fails for any package
The extraction process is logged and observable via admin tools
Schema diff reporting: when an extraction produces changes, the system logs what changed (new activities, removed properties, type changes)
Proposed approaches (evaluate and choose)
Option A: .NET Reflection Microservice
Build a small .NET CLI tool or HTTP microservice that:

Accepts a NuGet package ID and version
Downloads the .nupkg from the feed
Extracts the DLL, loads it via reflection
Discovers all classes inheriting from Activity/CodeActivity/NativeActivity
Extracts properties with their types, attributes ([RequiredArgument], [Category]), and default values
Returns a JSON schema matching our ActivityDef format
Pros: Most accurate, mirrors how Studio itself discovers activities. Cons: Requires .NET runtime in the deployment environment, adds infrastructure.

Option B: NuGet Package Metadata + Documentation Scraping
Download the .nupkg, extract the XML documentation file (if present — many UiPath packages include XML docs)
Parse the XML docs for class/method/property documentation
Cross-reference with the DLL's metadata tables (PE header parsing in pure JS — limited but possible for type names)
Supplement with UiPath's public API docs where available
Pros: No .NET runtime needed, runs in Node.js. Cons: XML docs aren't always present, PE parsing gives limited type info, less reliable.

Option C: LLM-Assisted Schema Extraction
On version change, fetch UiPath's release notes and documentation pages for the package
Use an LLM to extract structured schema changes from the documentation
Validate extracted schemas against the previous version's schema for sanity
Human-in-the-loop approval for significant changes (new activities, removed properties)
Pros: Zero infrastructure, leverages existing LLM capabilities. Cons: Less reliable, depends on documentation quality, potential for hallucination.

Option D: Hybrid — .NET CLI + Node.js Orchestration
Include a pre-built .NET CLI tool as a binary in the repo (self-contained .NET publish)
Node.js downloads the .nupkg, calls the CLI tool to extract schemas
CLI tool returns JSON, Node.js validates and integrates
Pros: Accurate like Option A, but simpler to deploy (single binary, no running service). Cons: Binary compatibility across deployment environments, binary size.

Tasks
Evaluate approaches — Assess each option against our deployment constraints (Replit environment, no Docker, Node.js runtime). Document tradeoffs and recommend one.
Prototype the chosen approach — Build a working proof-of-concept that can extract schemas for 3 representative packages (UiPath.System.Activities, UiPath.UIAutomation.Activities, UiPath.Excel.Activities).
Build schema diff engine — Compare extracted schemas against existing registry definitions. Report additions, removals, and modifications. Apply sanity thresholds (reject if >50% of activities disappear).
Integrate with metadata refresh cycle — When refreshGeneration() detects a version change for a package, trigger schema re-extraction for that package. Update activity-catalog.json with new schemas.
Add fallback and observability — If extraction fails, preserve existing registry schemas. Log extraction results, schema diffs, and failures to admin-visible logs.
Add tests — Validate extraction accuracy against known schemas for core packages.
