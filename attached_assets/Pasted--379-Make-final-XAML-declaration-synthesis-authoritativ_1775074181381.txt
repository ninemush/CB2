#379 - Make final XAML declaration synthesis authoritative before Studio Resolution Smoke Test

What & Why
Package generation is failing with Studio Resolution Smoke Test blocking errors because the final emitted XAML does not consistently contain the declarations required by the activities that are actually present in the file.

The current failures are not limited to a single package family. They include:
- ui:-prefixed infrastructure activities such as Comment, LogMessage, InvokeWorkflowFile, GetTransactionItem, and SetTransactionStatus
- uds:-prefixed Data Service activities such as CreateEntity and QueryEntity

Today, resolvability is spread across several layers:
- base settings templates in xaml-compliance.ts
- embedded workflow templates such as the ReFramework/state-machine Main.xaml template in xaml-generator.ts
- dynamic injection passes such as injectDynamicNamespaceDeclarations and runAuthoritativeNamespaceInjection
- the final Studio Resolution Smoke Test in package-assembler.ts

These layers do not currently agree by construction. The smoke test is correctly validating the final XAML, but the compiler is still relying on template assumptions and regex insertion paths rather than deriving the final required declarations from the actual final activity set in the file.

The fix is to make final declaration/header synthesis authoritative before the smoke test runs. Template parity fixes are included, but they are supporting fixes, not the main mechanism.

Done looks like
Before the Studio Resolution Smoke Test runs, the compiler inspects each final emitted XAML file and guarantees that every activity actually present in that file has the required:
- project dependency package
- xmlns declaration
- AssemblyReference entry
- NamespacesForImplementation import

This guarantee covers at minimum both:
- ui:-prefixed infrastructure activities
- uds:-prefixed Data Service activities

REFramework/state-machine workflows no longer fail due to missing UiPath.System.Activities / UiPath.UIAutomation.Activities declarations

Workflows using Data Service activities no longer fail due to missing UiPath.DataService.Activities declarations

The smoke test passes for the current failing scenario because the pre-smoke-test authoritative declaration synthesis now injects whatever the smoke test would otherwise reject

Regression tests validate both:
- a REFramework/state-machine workflow path
- a workflow containing uds/Data Service activities

Out of scope
Refactoring the entire multi-layer injection architecture
AI enrichment quality improvements
New activity support
Changes to emission approval policy
Changes to DHG or baseline-first compiler behavior

Tasks
Add an authoritative pre-smoke-test declaration synthesis pass in package-assembler.ts. This pass must inspect the final emitted XAML activity set for each file and derive the required dependency package, xmlns declaration, AssemblyReference entry, and NamespacesForImplementation import for every activity family present. If any required declaration is missing, inject it before the smoke test runs.

Expand the authoritative synthesis logic to cover both ui and uds activity families explicitly. Do not rely on generic prefix-level guessing alone where multiple packages can share the same prefix behavior. Resolution should be activity-level or activity-family-level first, using the registry/catalog mappings already present.

Patch the supporting templates for parity: add UiPath.System.Activities and UiPath.UIAutomation.Activities (Windows only where appropriate) to the UIPATH_VB_SETTINGS and UIPATH_CSHARP_SETTINGS base templates in xaml-compliance.ts, and add the same required assembly references to the embedded ReFramework/state-machine Main.xaml template in xaml-generator.ts. Include UiPath.DataService.Activities parity where those activity families are present in the generated file set. These template fixes are supporting parity fixes, not the sole enforcement mechanism.

Update injectDynamicNamespaceDeclarations and runAuthoritativeNamespaceInjection so they use the same authoritative resolution function as the pre-smoke-test synthesis pass. Remove or reduce divergence where one layer relies on template assumptions and another relies on activity scanning.

Add instrumentation/logging around the authoritative synthesis pass showing, per file:
- activities detected
- required packages/assemblies/namespaces/xmlns derived
- declarations injected
- declarations already present
This logging is for validation and debugging, not the primary fix.

Add regression tests:
- generate a REFramework/state-machine package and assert Main.xaml and Process.xaml contain the required AssemblyReference entries and namespace imports for ui infrastructure activities
- generate a workflow using uds:CreateEntity and uds:QueryEntity and assert UiPath.DataService.Activities assembly references and namespace imports are present
- run the Studio Resolution Smoke Test in the test path and assert it passes for both scenarios
