#430 - Package-DHG Parity Producer Alignment

What & Why
Task #429 introduced a fatal parity check between archived workflows and DHG workflow status. The latest run now fails with a Package-DHG Parity fatal divergence, where DHG says a workflow is a Stub but the archived content is not a stub. This is useful because it proves the pipeline still has multiple competing views of workflow truth.

The problem is no longer silent corruption. The problem is inconsistent status production across stages.

This task aligns the producers of:
- archived workflow content
- workflow status classification
- DHG workflow analysis/status
so they all operate from the same final archive-bound XAML set and the same final stub/non-stub classification rules.

Goals
1. Make archive content and DHG status derive from the same final workflow bytes.
2. Ensure stub classification is decided after final mutation/canonicalization, not before.
3. Eliminate Package-DHG parity mismatches for stub vs non-stub status.
4. Preserve fatal parity enforcement if divergence still occurs.

Done looks like
- A single authoritative workflow-status classifier runs on the final archive-bound XAML set.
- DHG uses that classifier output rather than independently inferring stub/non-stub status from an earlier or different representation.
- Archive parity and DHG parity both reference the same content hashes and workflow identities.
- Package-DHG parity mismatches disappear unless there is a genuine bug.
- If a mismatch still occurs, diagnostics identify exactly which stage produced the conflicting status.

Out of scope
- Broad catalog/filter alignment
- New canonicalization concepts
- Business-logic repair
- Reworking decomposition strategy

Tasks

1. Identify the current split
Trace where stub/non-stub status is currently decided for:
- archive content
- DHG workflow analysis
- any outcome/final-quality reporting
Document the stage and input source each one uses.

2. Create one authoritative workflow-status classification step
Implement a single classifier that runs on the final archive-bound XAML entries after all canonicalization/mutation steps and before DHG generation.
It must produce, per workflow:
- file
- content hash
- status (stub / non-stub / blocked / malformed / generated-but-unwired as applicable)
- rationale
- classifier version

3. Make DHG consume the authoritative classifier output
Remove or bypass any separate DHG-local stub inference that uses earlier representations or different heuristics.
DHG must use the authoritative post-canonicalization classification results.

4. Make parity compare the same identities and bytes
Ensure Package-DHG parity compares:
- same normalized file names
- same final content hashes
- same classifier status outputs
not stage-local approximations.

5. Add debug artifact
Emit a compact artifact such as:
`workflowStatusParity`
with, per workflow:
- file
- finalArchiveHash
- dhgInputHash
- archiveStatus
- dhgStatus
- identicalContent
- identicalStatus
- divergenceReason

6. Add regression tests
Cover:
- workflow remains stub after final mutation and DHG agrees
- workflow becomes non-stub after final mutation and DHG agrees
- workflow malformed/blocked and both sides agree
- parity failure when DHG is intentionally fed stale pre-mutation content
- normalized file naming edge case

Relevant files
- server/package-assembler.ts
- server/final-artifact-validation.ts
- server/uipath-pipeline.ts
- DHG generation / workflow analysis modules
- tests under server/__tests__/
