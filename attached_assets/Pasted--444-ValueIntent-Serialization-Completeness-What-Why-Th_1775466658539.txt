#444 - ValueIntent Serialization Completeness

What & Why
The XAML pipeline's ValueIntent system (structured expression objects like {"type":"vb_expression","value":"..."}) is leaking unserialized JSON objects into final XAML output in two patterns:

1. JSON objects in executable attributes
ValueIntent objects appear verbatim in Condition, Value, and Expression attributes instead of being resolved to VB.NET expression strings. The pre-archive assertion json_object_in_executable_attribute catches these but the pipeline stubs the entire workflow instead of resolving them.

2. Object payloads in Assign.To/Assign.Value slots
<Assign.To> elements contain <OutArgument x:TypeArguments="x:String">Nothing</OutArgument> instead of a valid variable reference. The pre-archive assertion object_payload_in_target_value_slot catches these.

In the V21 verification bundle, these two patterns killed 5 of 7 stubbed workflows (GetBirthdays, ResolveContact, GenerateAndValidateDraft, HandleHumanReview, plus GenerateAndValidateDraft again for the JSON attribute leak). This is now the #1 pipeline failure mode after the malformed-quote fix landed in Task #443.

Done looks like
- No ValueIntent JSON objects ({"type":"vb_expression",...} or {"type":"expression",...}) survive into final XAML attributes. They are all resolved to their VB.NET expression string before emission.
- Assign.To targets always contain a valid variable reference (bracket-wrapped variable name), never Nothing or a raw JSON object.
- If a ValueIntent cannot be resolved due to missing data, the pipeline emits a safe fallback variable name (for example obj_REVIEW_UnresolvedTarget) and records a handoff remediation instead of emitting Nothing.
- Fallback variable creation for unresolved Assign.To targets is deterministic, the variable is declared in the appropriate workflow scope, and the remediation entry includes provenance.
- In debug/test mode, the pipeline fails hard if a fallback variable is emitted without a matching declaration in the same workflow.
- The json_object_in_executable_attribute and object_payload_in_target_value_slot pre-archive assertions still exist as safety nets but fire zero times on the BirthdayGreetingsV26 test case.
- The final-pass guard is idempotent. If a ValueIntent has already been resolved to a valid VB.NET expression string, the guard leaves it unchanged without re-wrapping, double-quoting, or otherwise altering the emitted expression.
- Existing regression tests continue to pass.

Out of scope
- The InvokeWorkflowFile.Arguments prefix mismatch
- The mail Body child-element issue
- Changes to the pre-archive assertion detection logic itself

Tasks
1. Trace every code path in resolvePropertyValue, buildExpression, and resolveAssignTemplate where a ValueIntent object can reach XAML output without being converted to a string.
2. Add a final-pass guard in the expression builder that detects any remaining JSON-shaped strings ({"type":...}) and either resolves them or replaces them with a safe placeholder expression.
3. Fix resolveAssignTemplate so that when toVarName resolves to Nothing or an empty/invalid value, the pipeline generates a deterministic review-flagged variable name, declares it in the workflow's variable scope (via _lateDiscoveredVariables or equivalent), and records a handoff remediation with provenance (source workflow, activity display name, original unresolved value).
4. In debug/test mode, assert that every emitted fallback variable has a matching declaration in the same workflow.
5. Add a pre-emission sweep in the activity template resolver that catches any attribute value matching the ValueIntent JSON pattern and runs it through buildExpression before it reaches XAML output.
6. Ensure the final-pass guard is idempotent. Already-resolved VB.NET expressions must pass through unchanged.
7. Add regression tests covering:
   - ValueIntent object in a Condition attribute
   - Assign with no valid To target
   - Nested ValueIntent in a complex expression
   - Already-resolved VB.NET expressions (for example str_Name, int_Count + 1, String.Format(...)) pass through the final-pass guard completely unchanged
