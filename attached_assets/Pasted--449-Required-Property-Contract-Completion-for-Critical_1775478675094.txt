#449 - Required Property Contract Completion for Critical Activities

What & Why
After the recent fixes to quote safety, ValueIntent serialization, invoke dotted-tag prefix consistency, mail Body child-element handling, and auto-wiring diagnostics, the dominant remaining blockers are now concentrated in required runtime properties that are still being lost between spec, lowering, and final XAML.

Recent benchmark runs show:
- BirthdayGreetingsV26 still fails with Send Gmail Message missing required property "Body"
- BirthdayGreetingsV26 still fails with Update Entity missing required property "EntityObject"
- these failures survive late enough to keep the package structurally invalid even though the broader graph problem has narrowed

This is no longer just a mail-family issue. It is a platform-level contract preservation problem:
- required runtime properties exist upstream or should be derivable during lowering
- but they do not reliably survive into the final executable activity node

This task must not invent business values. It must preserve required property bindings when they already exist upstream, derive them only when deterministically available from existing structured intent, and fail earlier with explicit diagnostics when they cannot be satisfied.

Done looks like
- Required runtime properties for critical activities survive from spec/lowering into final XAML.
- Send Gmail Message.Body is present in final executable form when upstream intent provides it.
- Update Entity.EntityObject is present in final executable form when upstream intent provides it.
- Missing required critical properties no longer survive as late opaque structural invalidity when they were already available upstream.
- If a required property cannot be deterministically preserved or derived, the pipeline emits an explicit structured defect earlier with provenance.
- The benchmark BirthdayGreetingsV26 case no longer fails for missing Gmail Body or missing Update Entity EntityObject.

Out of scope
- Auto-wiring framework redesign
- Broad dependency-family redesign
- Mail family source-of-truth redesign
- Business-logic invention
- Download behavior for handoff-only packages

Tasks
1. Trace required-property loss on the active path
Instrument the active path from structured spec to final XAML for critical activities, specifically at least:
- GmailSendMessage.Body
- UpdateEntity.EntityObject

For each required property, record:
- whether it existed in the source spec
- whether it existed after lowering
- whether it existed after compliance/normalization
- whether it existed in final archive-bound XAML
- where it was lost if missing

2. Add required-property contract preservation
At the critical-activity lowering / template resolution boundary, preserve required property bindings into the final executable node whenever they already exist upstream.

3. Add deterministic derivation only where justified
If a required property is missing at emission time but can be deterministically derived from existing structured intent already present in the workflow, derive it and record provenance.
Do not invent business values.

4. Add early explicit failure for unsatisfied required contracts
If a required property for a critical activity cannot be preserved or deterministically derived, emit a structured contract-completeness defect earlier instead of letting the workflow drift into late structural invalidity without explanation.

5. Emit consolidated diagnostics
Add:
criticalActivityContractDiagnostics

Per activity include:
- workflow/file
- activityType
- requiredProperty
- presentInSpec
- presentAfterLowering
- presentAfterCompliance
- presentInFinalXaml
- derivationApplied
- derivationSource
- lossStage
- rootCauseCategory
- remediationHint

Summary counts:
- totalCriticalActivitiesChecked
- totalRequiredPropertiesSatisfied

- totalRequiredPropertiesDerived
- totalRequiredPropertiesLost
- totalEarlyContractFailures

6. Add benchmark-focused regression tests
Cover:
- Gmail Body present upstream survives to final XAML
- Update Entity EntityObject present upstream survives to final XAML
- deterministically derivable required property is preserved with provenance
- non-derivable required property fails early with explicit diagnostics
- no business-value invention occurs
