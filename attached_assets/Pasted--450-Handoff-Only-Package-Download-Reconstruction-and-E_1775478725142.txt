#450 - Handoff-Only Package Download Reconstruction and Empty-Buffer Guard Fix

What & Why
Users in handoff-only scenarios cannot download package artifacts even when the pipeline successfully built package content earlier and cached XAML entries still exist.

Observed behavior:
- pipeline builds a package buffer during the run
- final validation marks the result structurally_invalid / non-viable
- the pipeline zeroes out the package buffer for non-viable output
- the Developer Handoff Guide remains available
- the download route then fails early on an empty-buffer guard before it can attempt reconstruction from cached XAML entries

If this is correct, there are two product bugs:
1. an early empty-buffer guard short-circuits the download path before reconstruction logic can run
2. handoff-only packages are made completely undownloadable even though they should remain available for developer remediation with explicit warnings

This task must not mislabel invalid packages as deployment-ready. It must allow reconstruction/download for handoff-only artifacts while preserving the existing readiness warnings.

Done looks like
- The download route does not fail prematurely just because packageBuffer is empty if reconstructable cached XAML entries exist.
- Handoff-only / structurally invalid packages can still be downloaded as developer-remediation artifacts with explicit warnings.
- Reconstruction from cached XAML entries is attempted before returning an empty-buffer failure.
- Truly unreconstructable cases still fail honestly with a clear reason.
- UI/API surfaces continue to distinguish deployment-ready packages from handoff-only downloadable artifacts.

Out of scope
- Changing package viability rules
- Making structurally invalid packages appear deployment-ready
- Compiler/lowering fixes unrelated to download behavior

Tasks
1. Remove or defer the premature empty-buffer short-circuit
In the download route, ensure the early packageBuffer-empty guard does not return before reconstruction from cached XAML entries is attempted.

2. Preserve handoff-only downloadability
If final validation marks a package structurally_invalid or handoff-only, allow the system to return a downloadable reconstructed artifact when cached XAML entries are available, with explicit warning metadata.

3. Keep readiness semantics honest
Do not treat reconstructed handoff-only downloads as viable/deployment-ready packages.
They must remain clearly marked as handoff-only / not deployment-ready.

4. Emit download-path diagnostics
Add:
handoffDownloadDiagnostics

Per request include:
- runId / packageId
- packageBufferPresent
- cachedXamlEntriesPresent
- reconstructionAttempted
- reconstructionSucceeded
- failureReason
- returnedArtifactType (viable_package | reconstructed_handoff_package | none)

5. Add regression tests
Cover:
- viable package downloads normally
- empty packageBuffer with cached XAML entries reconstructs successfully
- handoff-only package remains downloadable with warnings
- unreconstructable package fails with clear reason
- readiness state remains not-ready for reconstructed handoff downloads
