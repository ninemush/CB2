#455 - Symbol Authority and Variable Declaration Completion

What & Why
Recent package runs show that the dominant remaining execution-blocking defect class is undeclared variables and arguments reaching quality gate and final validation. This is now showing up at high volume across complex flows, especially Coupa-style use cases.

The current pipeline does attempt symbol discovery before emission, but the evidence shows it is still incomplete:
- many variables used in executable expressions are never declared
- unprefixed semantic names are not being handled reliably
- discovery is still too dependent on prefix-based heuristics
- control-flow and nested-expression locations are still leaking undeclared symbols into final XAML
- downstream repair often substitutes Nothing, which avoids crashes but produces non-functional workflows

This is not just an LLM quality issue. It is a compiler authority issue. The pipeline must become authoritative about which symbols are referenced and which declarations must exist before quality gate.

This task must not build a second symbol framework. It must strengthen and complete the existing symbol discovery and declaration path so that variables and arguments referenced in emitted expressions are declared in the correct scope before quality gate.

Done looks like
- Executable expressions in final XAML no longer reference undeclared variables or arguments for the benchmark use cases.
- Variables referenced in activity properties, control-flow conditions, catch/finally bodies, loop constructs, and nested structured values are declared before quality gate runs.
- Unprefixed semantic variable names are handled when there is clear evidence they are true workflow symbols rather than literals or member/property access.
- The symbol scope enforcer no longer needs to replace large numbers of missing variables with Nothing.
- Quality gate undeclared-variable defects drop dramatically on the benchmark use cases.
- Existing regression tests continue to pass.

Out of scope
- Workflow invoke contract serialization and x:Members generation across workflows
- REFramework wiring changes
- Raw quote repair changes
- Namespace prefix authority for activity tags
- Prompt engineering to reduce variable mistakes at the LLM layer
- Inventing business values or silently guessing missing symbols without evidence

Tasks

1. Trace the live symbol discovery and declaration path
Trace how symbols are currently discovered, registered, scoped, and emitted through:
- pre-emission walk
- activity template resolution
- variable/argument registration helpers
- symbol scope enforcement
- quality gate undeclared-variable checks

Use the current benchmark failures as evidence targets.

2. Expand symbol discovery to all executable expression locations
Ensure the existing discovery walk covers at minimum:
- activity property expressions
- If / While / Switch conditions
- ForEach / ParallelForEach values and bodies
- Try / Catch / Finally bodies
- Assign targets and values
- Invoke argument bindings
- nested object and array property values
- expressions embedded in ValueIntent objects after lowering

3. Add unprefixed symbol detection with evidence gate
Extend discovery so it can identify likely workflow symbols even when they do not use strict prefixes like str_, int_, bool_, obj_, in_, out_, io_.

Constraints:
- only treat an unprefixed token as a symbol when there is strong syntactic evidence it is a standalone workflow variable or argument reference
- do not treat string literals, enum members, property access chains, method names, type names, or namespace identifiers as workflow symbols
- if evidence is ambiguous, do not silently declare it. Prefer an explicit diagnostic over guessing

4. Make symbol declaration authoritative before quality gate
Ensure that once a symbol is discovered and classified as a workflow variable or argument, the correct declaration is emitted into the right scope before quality gate runs.
Do not rely on downstream Nothing substitution as the normal path.

5. Preserve scope correctness
Ensure declarations land in the correct workflow scope and do not create duplicates or invalid shadowing.
If the same discovered symbol appears with conflicting inferred categories or incompatible types, record an explicit symbol conflict defect instead of guessing.

6. Add lightweight symbol diagnostics
Add or extend one existing diagnostics surface so benchmark runs can show:
- discovered symbol
- inferred category (variable or argument)
- inferred type
- declaration emitted or not
- scope location
- conflict or ambiguity reason if declaration was withheld

Keep this lightweight. Do not create a large parallel diagnostics framework unless necessary.

7. Add focused regression tests
Cover:
- prefixed symbol discovery still works
- unprefixed standalone workflow variables are discovered and declared
- control-flow condition symbols are discovered and declared
- nested structured property symbols are discovered and declared
- ambiguous tokens are not silently declared
- duplicate/conflicting symbol evidence produces explicit conflict defects
- quality gate no longer flags benchmark-style declared symbols as undeclared
