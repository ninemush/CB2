#458 - Workflow Interface Derivation and Spec-Level Contract Declaration

What & Why
The latest benchmark run shows that pre-emission workflow contract resolution cannot fully work because many invoked business workflows still have effectively empty spec-level interfaces. The tree assembler can only emit correct InvokeWorkflowFile bindings if the workflow specs themselves already declare the callee contract.

Current evidence shows the deeper issue is upstream of invoke emission:
- many invoked workflows still have targetDeclaredArguments = []
- pre-emission contract resolution therefore has no authoritative callee truth to apply
- post-emission contract validation still reports mismatches because the workflow interface was never made explicit in the spec layer
- 457 improved the assembly-side contract path, but the spec layer still does not reliably define workflow interfaces for business workflows

This task must not add another post-emission repair pass. It must make workflow interfaces explicit before tree assembly by deriving, validating, and attaching workflow argument declarations at the spec level.

Goal
Make spec-backed workflow interfaces authoritative enough that the tree assembler can build a meaningful pre-emission contract map for invoked workflows.

Done looks like
- Invoked business workflows in the benchmark use cases no longer arrive at tree assembly with empty or missing arguments arrays when there is clear spec evidence for their interfaces.
- WorkflowSpec.arguments is populated for invoked workflows from authoritative upstream evidence available before assembly.
- The pre-emission contract map built by the assembler has real callee contract truth for benchmark business workflows.
- InvokeWorkflowFile emission can use those spec-backed contracts on first pass.
- Post-emission contract-integrity defects caused by empty callee contracts drop materially on benchmark runs.
- No silent invention of unsupported workflow interfaces occurs.
- Existing post-emission contract validator and canonicalizer remain as safety nets.

Out of scope
- Rewriting invoke-binding-canonicalizer.ts or workflow-contract-integrity.ts into new primary systems
- Broad tree assembler contract emission changes already covered by 457
- Raw quote repair changes
- Activity schema timing fixes
- Expression authority fixes
- Prompt engineering
- Silently guessing interfaces when upstream evidence is ambiguous or conflicting

Design rules

Spec-level authority only
The workflow interface used for pre-emission contract resolution must come from pre-assembly sources only. Do not derive it from emitted XAML. Do not reuse buildWorkflowContracts from workflow-contract-integrity.ts for this purpose.

Evidence gate
Only declare a workflow argument in WorkflowSpec.arguments when there is explicit enough upstream evidence for:
- argument name
- direction
- usable type signal

Allowed evidence sources may include:
- invoke bindings from upstream callers
- explicit workflow spec metadata already present
- existing naming convention evidence such as in_/out_/io_
- stable type inference rules already in the codebase

If evidence is incomplete or conflicting, do not silently synthesize the interface. Record an explicit interface derivation diagnostic instead.

Conflict handling
If multiple callers imply conflicting directions or incompatible types for the same workflow argument, record a conflict diagnostic and do not auto-declare that argument until the conflict is resolved.

Framework and unresolved workflow behavior
If a target workflow is framework-owned or outside the available spec set, do not invent a business workflow interface for it. Fall through to the current behavior and let existing downstream safety nets remain in place.

Tasks

1. Trace where WorkflowSpec.arguments should be populated
Identify the pre-assembly path where workflow specs are created, enriched, merged, or normalized, and determine why invoked business workflows still reach assembly with empty or insufficient arguments arrays.
Focus on the actual spec lifecycle before package assembly.

2. Add spec-level interface derivation before tree assembly
Introduce a pre-assembly pass that inspects the full workflow spec set and derives candidate workflow interfaces for invoked workflows using only allowed evidence.
This pass must populate or augment WorkflowSpec.arguments for target workflows before the tree assembler builds its pre-emission contract map.

3. Reuse existing naming and type inference conservatively
Use existing direction conventions and existing type inference utilities where available.
Do not invent new broad inference rules if current utilities are sufficient.
If evidence is weak, prefer an explicit diagnostic over a guessed declaration.

4. Add explicit diagnostics for incomplete or conflicting interface derivation
For each workflow where interface derivation is attempted, surface:
- workflow name
- derived arguments
- evidence source
- arguments withheld due to ambiguity
- conflicts across callers
Keep this lightweight and attach it to an existing reporting surface if possible.

5. Feed derived interfaces into existing pre-emission contract resolution
Ensure the resulting WorkflowSpec.arguments data is available to the 457 contract-map path before per-workflow assembly begins.
Do not duplicate contract resolution logic in a second place.

6. Add focused regression tests
Cover:
- invoked workflow with empty initial arguments array gets interface populated from caller evidence
- direction derived from in_/out_/io_ evidence
- type inferred from existing utilities where evidence is sufficient
- conflicting caller evidence produces explicit diagnostic and no silent declaration
- unresolved or framework workflow falls back without invented interface
- pre-emission contract map now contains business workflow contracts that were previously empty
- post-emission contract-integrity failures drop for benchmark-style empty-interface cases

Primary files to modify
- server/uipath-pipeline.ts
- server/workflow-spec-types.ts
- server/package-assembler.ts
- server/workflow-tree-assembler.ts
- the actual spec construction / enrichment / normalization files where WorkflowSpec.arguments is produced before assembly

What this does not touch

- prompt changes
- quote escaping systems
- broad schema enforcement
- broad expression builder redesign

Success measure
On benchmark runs, invoked business workflows that previously showed empty targetDeclaredArguments should now have spec-backed contracts before assembly, enabling 457-style pre-emission contract resolution to act on real callee truth. Contract-integrity defects caused by missing or empty business workflow interfaces should drop materially.
