#460 - Typed Property Object Boundary Enforcement and Pre-Serialization Normalization

What & Why
The latest package run failed earlier than recent structurally-invalid runs: packaging was blocked at the Emission Gate with 21 normalization invariant violations of the same class, all reporting that a typed property object leaked into XAML.

This is not just an LLM quality problem. The deeper issue is that the pipeline still lacks a hard normalization boundary between:
- typed/structured property objects
- scalar attribute-safe values
- child-element values with known schema shape

A strong pipeline should never allow an unresolved typed property object to reach final XAML serialization. If a property is still object-shaped at that point, the pipeline must have already done one of the following:
- lowered it to a valid scalar value
- routed it into a known child-element emission path
- rejected it explicitly with a structured diagnostic before generic XAML serialization begins

The latest run also exposed an upstream schema edge case:
- a workflow step returned an Arguments property where a recognized typed property object had value: undefined
- this produced a validation failure on one attempt
- after retry, object-shaped values still leaked into XAML and were blocked by the Emission Gate

This task must harden the typed-property normalization boundary before XML serialization. It must not weaken the Emission Gate. The gate is correctly blocking invalid emission. The fix is to make the pipeline normalize or reject typed objects earlier and more consistently.

Done looks like
- Typed property objects no longer leak into emitted XAML on benchmark runs.
- Emission Gate typed-property-object invariant violations drop materially or disappear on benchmark runs.
- Structured property values are classified before serialization as one of:
  - scalar attribute-safe value
  - child-element value with known schema shape
  - unsupported structured object with explicit diagnostic/block
- Recognized typed objects with missing `value` fields are normalized or rejected consistently instead of failing unpredictably.
- Generic XAML emission paths no longer attempt to serialize raw typed property objects.
- The Emission Gate remains a hard safety net, but it is no longer the primary place where typed-object leakage is discovered.

Out of scope
- Weakening or bypassing the Emission Gate
- Broad workflow contract redesign
- Graph connectivity fixes
- Broad required-property timing redesign
- Prompt engineering
- Rewriting all expression logic
- Converting unsupported structured objects into guessed string values
- Silent coercion of malformed typed property objects into arbitrary literals

Design rules
Hard boundary before serialization
Before any property reaches generic XML/XAML serialization, the pipeline must classify it into exactly one of:
1. scalar value safe for attribute emission
2. structured value with known child-element emission shape
3. unsupported or incomplete structured value that must be blocked or diagnosed

No raw typed object may cross that boundary.

No silent invention
If a typed property object is incomplete, ambiguous, or unsupported, do not invent missing meaning just to make it serialize. Prefer an explicit diagnostic and controlled block over guessed semantics.

Schema-aware routing
If a property’s schema requires child-element emission, do not try to flatten it into an attribute-safe scalar. Route it to the correct structured emission path or reject it if the shape is unsupported.

Preserve successful scalar paths
Do not regress already-valid scalar serialization paths. If a property is already correctly normalized to a scalar value, this task must leave it alone.

Keep the gate strict
Do not relax the Emission Gate invariant for typed-property-object leakage. The goal is to stop the leak upstream, not to tolerate it downstream.

Tasks
1. Trace the typed-property-object leak path
Use the latest benchmark failure as evidence and trace where typed property objects are:
- accepted by input schemas or preprocessors
- preserved through normalization
- passed into tree assembly
- passed into generic XAML/property serialization
- finally detected by the Emission Gate

Specifically inspect:
- property input schema / preprocessing for typed objects with missing `value`
- workflow-tree-assembler typed property normalization
- xaml-generator property serialization
- emission-gate typed-property-object invariant checks

2. Harden property preprocessing for incomplete typed objects
Review the property input preprocessing path and make typed objects with missing `value` fields normalize or fail consistently.
Important:
- handle the case where an object matches a recognized typed-property pattern but contains `value: undefined`
- do not let such objects drift through as half-valid structured values
- if safe normalization is not possible, emit a clear diagnostic and prevent the malformed shape from reaching later stages

3. Add explicit typed-property classification before serialization
Introduce or strengthen a normalization step before generic XAML emission that classifies each property value into:
- scalar attribute-safe
- child-element structured
- unsupported structured object

This classification must happen before any final XML attribute or element interpolation.

4. Block raw typed objects from generic emission loops
Update generic property serialization so it refuses to serialize unresolved typed property objects.
If a raw typed object arrives at this point, treat it as a normalization failure with explicit diagnostics rather than attempting raw interpolation.

5. Route supported structured values correctly
Where structured values are valid and schema-supported, route them into the correct emission shape instead of flattening them incorrectly.
This includes:
- typed child-element properties
- supported structured argument/property forms already recognized by the pipeline

6. Add lightweight diagnostics
Extend an existing reporting surface lightly so benchmark runs can show:
- property classified as scalar / child-element / unsupported structured
- typed object normalized successfully or blocked
- source stage where malformed typed object was first detected
Keep this lightweight. Do not create a large parallel diagnostics framework.

7. Add focused regression tests
Add tests covering:
- recognized typed property object with `value: undefined` is handled consistently
- raw typed property object does not leak into generic XAML serialization
- supported structured property is routed to correct child-element shape
- unsupported structured property is blocked with explicit diagnostic
- scalar values still serialize unchanged
- Emission Gate typed-property-object invariant is no longer triggered for benchmark-style leaks

The pipeline should either normalize structured properties correctly before emission or reject them explicitly before generic XAML serialization begins.
