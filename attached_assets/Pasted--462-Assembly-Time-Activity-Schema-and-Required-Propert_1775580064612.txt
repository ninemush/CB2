#462 - Assembly-Time Activity Schema and Required-Property Authority

What & Why
Recent benchmark runs, especially the latest BirthdayGreetings run, show that the dominant remaining structural-invalidity class is now activity schema and required-property failure at assembly time rather than typed-property leakage.

The latest Birthday run shows repeated:
- ui:LogMessage missing required property "Message"
- Send SMTP / mail activity Body emitted in the wrong shape
- pseudo-XAML Body="[str_EmailBody]" patterns still reaching validation
- child-element vs attribute shape still being discovered too late

This means the pipeline still knows too much activity schema truth only after emission, through required-property-enforcer, quality gate, and post-assembly checks. A Studio-like builder would know the activity contract during assembly and emit the correct property shape on the first pass.

This task must move activity schema truth earlier so the assembler emits required properties, namespaces, and child-element vs attribute shapes correctly before post-emission enforcement.

Done looks like
- LogMessage activities no longer emit without Message on benchmark runs.
- Mail send activities no longer emit Body as pseudo-XAML attributes when schema requires child-element form.
- Required-property failures for benchmark use cases drop materially at source rather than being discovered mainly by post-emission checks.
- Dedicated template paths and generic emission paths obey the same activity schema truth.
- Existing required-property-enforcer and quality gate remain as safety nets, but they are no longer the primary place where these issues are discovered.

Out of scope
- Graph connectivity / unreachable workflow fixes
- Workflow contract/interface redesign
- Typed property object boundary work already covered by 460/461
- Prompt engineering
- Broad symbol-authority redesign
- Weakening post-emission validators

Design rules
1. Assembly-time schema authority
The activity catalog / registry must be consulted before emission so the assembler knows:
- which properties are required
- which properties must be child elements vs attributes
- which activity namespace/prefix is required
- which safe placeholder/remediation form is allowed when real business data is absent

2. No pseudo-XAML placeholders
If a property requires child-element shape, do not emit pseudo-XAML attributes like Body="[str_EmailBody]". Route it to the correct child-element emission path or block it diagnostically.

3. Dedicated-template parity
Dedicated templates (for example LogMessage, mail activities, data service activities, and similar custom branches) must obey the same schema truth as the generic path. No template-specific shortcuts that bypass required-property or child-element rules.

4. Honest placeholder policy
If a required property has no authoritative business value, do not invent logic. Only emit a safe placeholder when:
- the schema permits it
- the placeholder keeps Studio loadable
- a clear diagnostic / DHG remediation is recorded
If no safe placeholder form exists, block diagnostically rather than emitting an invalid activity.

Tasks
1. Trace activity schema truth through the current pipeline
Identify where the pipeline currently knows:
- required properties
- child-element vs attribute shape
- namespace ownership
- safe default/placeholder policy

Trace how this truth flows through:
- uipath-activity-registry.ts
- catalog/xaml-template-builder.ts
- workflow-tree-assembler.ts
- required-property-enforcer.ts
- uipath-quality-gate.ts

Use the latest Birthday run as the primary evidence target:
- repeated missing LogMessage.Message
- mail Body pseudo-XAML / wrong-shape emission

2. Make assembly consult schema truth before emission
Introduce or strengthen an assembly-time lookup so the emitter knows the required property contract of each activity before building XML.
This must apply to both:
- generic emission paths
- dedicated templates

3. Fix LogMessage required-property emission
Ensure LogMessage never emits without a Message property.
If an authoritative message value exists, emit it correctly.
If it does not exist, apply the honest placeholder policy:
- safe placeholder only if allowed
- otherwise explicit diagnostic/block
Do not emit structurally incomplete LogMessage activities.

4. Fix mail Body child-element emission at assembly time
Ensure mail Body is emitted in the correct child-element form when required by schema.
Do not rely on post-emission repair of Body="[...]" patterns.
If an authoritative body binding exists, emit the proper child-element shape.
If it does not, apply the honest placeholder policy rather than leaving the property absent or emitting pseudo-XAML.

5. Enforce schema parity across dedicated templates
Audit dedicated template branches in workflow-tree-assembler.ts for:
- LogMessage
- mail activities
- any other benchmark-hit activities with required-property failures
Route them through the same schema-aware decisions as the generic path.

6. Add lightweight diagnostics
Extend an existing reporting surface so benchmark runs can show:
- activity type
- required property missing upstream or present
- assembly-time decision taken
- child-element vs attribute shape selected
- placeholder emitted or blocked
Keep this lightweight. Do not add a new diagnostics framework.

7. Add focused regression tests
Add tests covering:
- LogMessage with valid Message emits correctly
- LogMessage with no authoritative Message does not emit as structurally incomplete
- mail Body emits as correct child-element shape
- pseudo-XAML Body="[...]" no longer reaches validation
- dedicated template and generic emission paths obey the same schema rules
- benchmark-style missing required-property cases are prevented at assembly time rather than first discovered post-emission

The pipeline should emit schema-correct activities on first pass rather than discovering missing required properties mainly after emission.
