#462 - Required-Property Failure Path Isolation for Birthday Benchmark

What & Why
The latest Birthday benchmark still fails structurally with repeated required-property defects, especially around:
- LogMessage missing Message
- mail send activity missing Body
- residual fragile defects after iterative correction

However, current code inspection suggests some relevant dedicated templates may already handle these properties in at least one path. That means the remaining failures are likely path-specific, not broad family-wide failures.

This task is not to implement assembly-time schema authority yet.
This task is to isolate the exact active failing path(s) for the benchmark and produce evidence-backed implementation guidance for the next narrow fix.

Output location
Write the full findings to:
docs/architecture/birthday-required-property-path-isolation.md

This file is the sole deliverable.

Done looks like
- The failing Birthday benchmark cases are traced to specific active code paths, not broad assumptions.
- For each repeated required-property failure class, the output identifies:
  - exact failing activity type
  - exact property missing or malformed
  - exact emitted XAML shape observed
  - exact template/emission path responsible
  - whether the failure came from dedicated template, generic emission path, fallback/remediation path, or post-emission mutation
- The output distinguishes confirmed evidence from code inference.
- The document ends with one recommended next implementation task, bounded to the proven failing path only.

Out of scope
- Implementing the fixes
- Broad assembly-time schema redesign
- Graph connectivity work
- Workflow contract work
- Typed-property-object boundary work
- Prompt engineering
- A broad audit of every activity family

Tasks
1. Use the latest Birthday benchmark artifacts as the evidence base.
For each repeated required-property failure in the run, collect:
- build log references
- DHG references
- quality gate / final validation references
- emitted XAML snippets when available

2. Isolate the exact failing path for repeated LogMessage defects.
Determine:
- whether the failure came from the dedicated LogMessage template
- whether it came from a generic property-emission path
- whether it came from a fallback/remediation mutation after initial emission
- whether the emitted Message was absent, empty, placeholder-only, stripped later, or malformed in shape

3. Isolate the exact failing path for repeated mail Body defects.
Determine:
- which activity family is failing (SMTP, Gmail, or another mail activity)
- whether the failing path used the dedicated mail template or a generic/fallback path
- whether the Body property was absent, emitted as pseudo-XAML, emitted in wrong child-element shape, or lost later

4. Distinguish upstream emission defects from downstream mutation defects.
For each confirmed failing case, identify the earliest stage where the property became wrong:
- assembly
- template resolution
- normalization/compliance
- required-property enforcer
- quality gate/final validation only

5. Bound the next implementation target.
End the document with exactly one recommended implementation task that:
- names the exact activity/path to fix
- names the primary files/functions to touch
- states what it will not touch
- avoids broad family-wide claims unless the evidence actually proves them

Important constraints
- Do not generalize from code structure alone.
- Mark each conclusion as either:
  - confirmed by benchmark evidence
  - inferred from code only
- Do not propose a broad multi-activity implementation task unless the evidence proves the same path is responsible across those activities.
