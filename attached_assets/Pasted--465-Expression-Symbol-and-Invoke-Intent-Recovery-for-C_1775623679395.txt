#465 - Expression, Symbol, and Invoke Intent Recovery for Complex Workflow Packages

What & Why
Recent complex benchmark runs show that the package outcome is still blocked by several independently fatal defect classes, but they are not fully independent. A large share of the blocking surface is generated upstream by missing structural intent in three connected areas:

1. structured JSON expressions are not fully lowered into valid VB.NET
2. variables referenced in expressions are not declared, so later stages degrade them to Nothing
3. InvokeWorkflowFile bindings often have valid XML form but missing semantic target argument identity

These three failures then cascade into downstream defects such as:
- If.Condition invalid/defaulted to False
- Assign.To = Nothing
- LogMessage.Message and other properties losing usable values
- broken argument bindings
- structurally incomplete activities

This task must not be a broad rewrite and must not be use-case-specific. It must add a narrow, evidence-backed recovery layer for these three connected upstream failure classes so that downstream validators receive real values and intent more often, instead of degraded placeholders.

Done looks like
- Structured JSON expression objects for benchmark-hit paths are lowered to valid VB.NET before required-property enforcement relies on them.
- Variables referenced in executable expressions are declared when evidence from expression context is strong enough.
- InvokeWorkflowFile bindings with missing semantic target names are repaired when caller/callee evidence is strong enough.
- Downstream defects such as Assign.To = Nothing, blocked If.Condition defaults, and Nothing-bound invoke arguments drop materially on complex benchmark runs.
- Ambiguous cases are blocked explicitly with diagnostics rather than guessed.

Out of scope
- Full graph connectivity redesign
- Broad required-property redesign across all activities
- Prompt engineering
- Weakening quality gate or final validation
- Workflow-name-specific hacks
- Broad family-wide template rewrites
- Typed-property-object boundary work already covered elsewhere

Design rules
1. Upstream recovery only
This task must focus on upstream structural-intent recovery, not downstream symptom cleanup.

2. Evidence-backed repair only
Repair only when evidence is strong enough from:
- expression structure
- local workflow context
- caller/callee contract evidence
- activity/property type expectations
If evidence is weak or conflicting, block diagnostically.

3. No generic placeholder substitution
Do not "fix" these cases by emitting generic defaults like Nothing, empty string, or False when those values are already known to cause downstream structural failures.

4. Shared-path only
The fixes must apply through shared code paths, not benchmark-specific workflow names.

Tasks
1. Freeze the benchmark evidence set
Use the latest Birthday and Coupa benchmark artifacts and DHG/final validation output.
Confirm the repeated blocking classes this task targets:
- structured JSON expression leaks
- undeclared variables from expression usage
- invoke bindings with missing semantic target identity
Do not broaden beyond what the evidence confirms.

2. Strengthen structured expression lowering
Improve the shared expression-lowering path so benchmark-hit structured JSON expression forms are lowered before required-property enforcement and final emission rely on them.
This includes the expression forms currently leading to blocked If/While conditions and downstream required-property degradation.

3. Add variable declaration recovery from expression context
Extend the shared declaration/symbol path so variables referenced in executable expressions are auto-declared when type/category/scope can be inferred strongly enough from:
- expression context
- nearby assignments/usages
- activity/property expectations
- surrounding workflow scope
If ambiguous, emit explicit diagnostic instead of guessing.

4. Add semantic invoke-binding recovery
For InvokeWorkflowFile calls where the XML shape is valid but argument identity is missing or bound to Nothing, use:
- caller-side expression evidence
- callee contract/interface evidence
- direction/type compatibility
to repair the binding when confidence is high.
If multiple plausible targets exist, block diagnostically instead of guessing.

5. Add lightweight diagnostics
Extend an existing reporting surface to show:
- expression lowered or not
- variable auto-declared or blocked
- invoke binding repaired or blocked
- evidence source used for the repair
No new diagnostics framework.

6. Add focused regression tests
Cover:
- benchmark-hit JSON expression forms lower correctly
- variable referenced in expression is declared when evidence is strong
- ambiguous variable case blocks instead of guessing
- invoke binding to Nothing is repaired when caller/callee evidence clearly aligns
- ambiguous invoke binding remains blocked with explicit diagnostic
- downstream symptom classes materially reduce in regression coverage

Success measure
Complex benchmark runs should show a material reduction in the upstream structural-intent failure classes and in the downstream symptom classes they currently generate, including:
- undeclared variables
- blocked If/While condition defaults
- Assign.To = Nothing
- invoke bindings with missing semantic target identity

The goal is not cosmetic defect reduction. The goal is to remove enough upstream fatal classes that benchmark packages can move meaningfully closer to viable output.
