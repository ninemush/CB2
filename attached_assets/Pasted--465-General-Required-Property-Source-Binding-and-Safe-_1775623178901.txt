#465 - General Required-Property Source Binding and Safe Fallback Authority

What & Why
Recent benchmark runs across multiple use cases show a repeated structural failure class: activities reach validation with required properties unresolved because the pipeline cannot consistently decide between:
- binding an authoritative upstream source
- emitting an allowed structural fallback
- explicitly blocking when neither is valid

This has appeared in different forms across runs, including repeated issues with:
- LogMessage.Message
- mail Body properties
- If.Condition
- additional required properties in shared activity families when upstream values are weak, empty, generic, or missing

The problem is not that each benchmark needs a custom patch. The problem is that the pipeline still lacks a reusable required-property decision authority under uncertainty.

This task must implement a general-purpose, evidence-backed decision layer for benchmark-hit required-property classes. It must not become a one-off fix for a single workflow or use case.

Done looks like
- The pipeline uses a consistent decision framework for benchmark-hit required properties:
  1. bind authoritative source if available
  2. otherwise emit an explicitly allowed structural fallback if policy permits
  3. otherwise block with a clear diagnostic
- Repeated required-property failures drop materially across benchmark runs without one-off use-case patching.
- Activities are no longer left structurally incomplete when a valid source or allowed fallback policy exists.
- Existing validators remain strict and continue acting as safety nets.

Out of scope
- Use-case-specific hacks
- Broad redesign of every activity family
- Graph connectivity fixes
- Workflow contract/interface redesign
- Typed-property-object boundary work
- Prompt engineering
- Weakening quality gate or final validation
- Inventing business logic where no safe fallback exists

Design rules
1. Generalizable only
Only include activity/property classes that are:
- confirmed by benchmark evidence
- repeated across multiple runs or use cases, or
- governed by a shared code path / activity class policy

2. Source first, fallback second, explicit block third
This decision order is mandatory.

3. No invalid generic defaults
Do not emit empty strings, False, or other generic placeholders where the activity/property contract or validator already rejects them.

4. Explicit fallback policy
For every included activity/property class, define:
- whether fallback is allowed
- the exact fallback value/shape allowed
- when block-with-diagnostic is required instead

5. No one-off benchmark logic
Do not special-case workflow names or use-case-specific artifacts.

Tasks
1. Freeze the evidence-backed target classes
Using the latest Birthday and Coupa benchmark evidence plus any completed isolation reports, identify the required-property classes that are truly general enough for this task.
Do not include any class that is only seen once or only in one workflow unless it is clearly part of a shared activity family/code path.

2. Trace the shared decision path
For each included class, trace:
- where the value is first sourced
- where generic/empty/invalid values are introduced
- where existing enforcer logic rejects or blocks them
- whether a shared code path already exists for value selection

3. Define the reusable decision policy
For each included class, define:
- valid source criteria
- allowed structural fallback if any
- explicit block condition if neither is valid

4. Implement the decision layer
Apply the policy in the shared value-selection/enforcement path rather than per-use-case fixes.

5. Add lightweight diagnostics
Show, for each included class:
- source found or not
- fallback applied or not
- block reason if unresolved

6. Add focused regression tests
Cover:
- valid source chosen when present
- allowed fallback used only when policy permits
- explicit block when neither valid source nor allowed fallback exists
- no generic invalid defaults emitted for included classes

Success measure
