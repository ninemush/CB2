#465 - Single Emission-Gate Blocker Isolation and Repair for Birthday Benchmark

What & Why
The latest Birthday benchmark no longer appears dominated by broad structural-invalid defect spam. Instead, the package build now fails earlier with a single Emission Gate contract violation.

That is a high-value signal:
- the simpler benchmark may now be close enough that one remaining emission-contract defect is the main blocker
- launching a broad structural-intent repair task before isolating this blocker would be premature and likely wasteful

This task must isolate the exact remaining Emission Gate blocker for the latest Birthday benchmark, identify the exact file/activity/property/path responsible, and implement a narrowly bounded fix if the root cause is confirmed. If the evidence shows the blocker is not yet specific enough for safe implementation, the task must stop after producing the isolation result and bounded fix recommendation.

Done looks like
- The exact single Emission Gate blocker in the latest Birthday benchmark is identified.
- The output names:
  - exact file
  - exact activity
  - exact property or contract slot
  - exact invariant/validator that blocked it
  - exact earliest stage where the value/shape became invalid
- The task distinguishes:
  - confirmed by benchmark evidence
  - inferred from code only
- If the root cause is narrow and proven, implement the bounded fix and add regression coverage.
- If the root cause is still ambiguous, do not guess. Stop after evidence isolation and write the bounded next fix recommendation.

Output location
Write the isolation findings to:
docs/architecture/birthday-single-emission-gate-blocker.md

If a code fix is implemented, update that document with:
- root cause
- fix applied
- regression test added
- remaining known blockers

Out of scope
- Broad required-property redesign
- Broad graph/connectivity redesign
- Broad workflow intent repair for complex packages
- Prompt engineering
- Weakening or bypassing the Emission Gate
- Multi-defect cleanup beyond the single confirmed blocker unless the exact same root cause is proven to drive them

Tasks
1. Freeze the evidence set
Use the latest Birthday benchmark artifacts for the run that failed with:
- exactly 1 Emission Gate contract violation
If the existing artifacts are incomplete, rerun the benchmark and collect:
- build logs
- emission gate output
- final validation output
- emitted XAML for the failing file(s)
- any DHG/output report produced before failure

2. Identify the exact blocker
Determine:
- the exact invariant name or violation class
- the exact file/workflow
- the exact activity and property/slot involved
- whether the blocker is:
  - required-property blank/incomplete
  - typed-object leakage
  - malformed invoke/contract binding
  - invalid child-element vs attribute shape
  - namespace/alias issue
  - another single invariant class

3. Trace the earliest failing stage
For the confirmed blocker, identify the earliest stage where the defect became wrong:
- spec/preprocessing
- tree assembly
- dedicated template dispatch
- generic emission path
- required-property enforcer
- compliance/normalization
- pre-gate canonicalization
- emission gate only

4. Decide whether the fix is safely bounded
If the blocker is narrow and clearly attributable to one active path, implement the bounded fix.
If the blocker is still ambiguous or appears to require a broad redesign, do not guess. Stop after isolation and recommendation.

5. Add focused regression coverage if a fix is implemented
Add one or more regression tests that reproduce the exact blocker and prove the fix.

Important constraints
- Do not generalize from earlier runs or from the Coupa benchmark.
- Do not broaden to a family-wide fix unless the evidence proves the same root cause.
- Mark each conclusion as either:
  - confirmed by benchmark evidence
  - inferred from code only
- If implementation proceeds, keep the blast radius minimal and tied to the proven path.
