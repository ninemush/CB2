# CB2-F06 - Workflow Reachability and REFramework Wiring Validation

What & Why
Generated UiPath packages can contain workflows that are syntactically present in the archive but not actually reachable from Main.xaml or correctly wired into the executable graph. This creates packages that look complete in the UI but are structurally incomplete in Studio or at runtime. Current quality checks focus on XAML validity, dependencies, and executable-path contamination, but they do not fully validate graph integrity across invoked workflows and REFramework state transitions.

This task adds graph-based workflow reachability and wiring validation so the package can distinguish between:
- workflows that merely exist on disk
- workflows that are actually reachable and correctly used in the executable graph

The validator must work for REFramework-style packages and general InvokeWorkflowFile-based packages. It must preserve build mode and artifact generation, but emit structured diagnostics plus a downgrade signal for downstream status derivation.

Done looks like
A dedicated workflow-graph validator analyzes the final packaged workflow set and constructs an executable graph rooted at Main.xaml.
It detects:
- orphaned workflows that are present in the package but unreachable from Main.xaml
- broken InvokeWorkflowFile references to missing or malformed target files
- decomposed workflows that were emitted but never wired into the actual graph
- REFramework state-machine wiring defects, including missing or inconsistent links among Init, GetTransactionData, Process, SetTransactionStatus, End Process, Dispatcher, Performer, or similarly named generated state workflows
- duplicate or ambiguous workflow references where multiple similarly named files could satisfy a call but no deterministic winner exists
- graph discontinuities where a workflow is referenced but cannot be parsed, loaded, or traversed further

The validator runs on the final post-repair, post-normalization XAML content actually written into the package archive, not on pre-repair intermediates.

Structured diagnostics are emitted as `workflowGraphDefects[]` with fields:
- file
- workflow
- defectType
- referencedFrom
- referencedTarget
- severity
- detectionMethod
- notes

A boolean downgrade signal such as `hasWorkflowGraphIntegrityIssues` is emitted alongside the defects array. The validator does not directly hardcode any specific package status. Downstream status derivation maps the signal into the current status model in force.

Build mode is preserved. Package ZIP and Developer Handoff Guide are still generated even when graph defects exist.

Regression tests cover:
- reachable workflow chains from Main.xaml
- orphaned helper workflows
- broken InvokeWorkflowFile target references
- REFramework wiring defects
- decomposed-but-unwired workflows
- clean graphs that should pass without false positives

Key invariants
- The graph validator is authoritative for executable reachability, but it does not replace dependency validation or executable-path validation.
- Only workflows intended to participate in execution should be considered for orphan detection. Support files, templates, docs, and clearly non-executable artifacts must be excluded.
- Main.xaml is the root of truth unless a package explicitly declares a different entrypoint in a supported way.
- The validator must use the final archive workflow set only.
- The validator must not mutate XAML. This is read-only validation.

Out of scope
- Changing XAML generation logic itself
- Repairing broken workflow references automatically
- Adding new package status values
- Replacing existing dependency or executable-path validators
- Deep semantic validation of business logic correctness inside otherwise reachable workflows

Tasks

1. Build the workflow graph validator module
Create `server/xaml/workflow-graph-validator.ts`.

Input:
- final XAML entries as `{ name: string; content: string }[]`
- any needed activity registry helpers or graph utilities

Output:
- `workflowGraphDefects: WorkflowGraphDefect[]`
- `hasWorkflowGraphIntegrityIssues: boolean`
- optional `workflowGraphSummary` with counts such as reachable workflows, unreachable workflows, broken references, ambiguous references

The validator must:
- identify executable workflow files in the final package
- parse `InvokeWorkflowFile` usages and equivalent workflow-invocation constructs
- build directed edges between workflows
- treat `Main.xaml` as the root entrypoint unless explicitly overridden by supported package metadata
- compute the reachable workflow set from the root
- flag executable workflows that are present but unreachable
- flag references to missing targets
- flag references to targets that exist but cannot be parsed/traversed
- flag ambiguous references if matching is not deterministic

2. Add REFramework-aware structural validation
Extend the validator with REFramework-aware checks.

Detect and report:
- missing expected state workflows when Main.xaml is clearly REFramework-like
- inconsistent dispatcher/performer wiring
- broken transitions where a generated state references a child workflow that is absent or unreachable
- decomposed transaction-processing workflows that exist but are not connected to the actual process path
- inconsistent naming or file linkage between generated REFramework components

This must not be a rigid filename-only checker. Use structure and invocation relationships first, then naming conventions as supporting heuristics.

3. Define executable workflow inclusion and exclusions
Implement a clear policy for which files participate in graph validation.

Include:
- Main.xaml
- invoked child workflows
- dispatcher/performer/state workflows
- decomposed helper workflows that are meant to execute

Exclude:
- templates
- docs
- DHG artifacts
- clearly non-executable support files
- files intentionally marked as stubs only if package logic already classifies them as non-executable support artifacts rather than executable workflows

If a file looks executable but is excluded, emit a low-severity note in diagnostics so the exclusion is explainable.

4. Integrate into final artifact validation
Wire the validator into `server/final-artifact-validation.ts` so it runs during `runFinalArtifactValidation` on the same final XAML entries used for packaging.

Add:
- `workflowGraphDefects`
- `hasWorkflowGraphIntegrityIssues`
- optional `workflowGraphSummary`

to `FinalQualityReport`.

In the status derivation section, feed `hasWorkflowGraphIntegrityIssues` into the existing degradation composite alongside the other degradation signals already present. Do not hardcode or reintroduce any legacy status value. Let the current package status derivation model in force handle the mapping.

Also wire the call site in `uipath-pipeline.ts` if needed to ensure the validator receives the final post-normalization XAML content after all repair and normalization passes have completed.

5. Emit structured defect categories and severities
Define `WorkflowGraphDefect` with at least these fields:
- `file`
- `workflow`
- `defectType`
- `referencedFrom`
- `referencedTarget`
- `severity`
- `detectionMethod`
- `notes`

Supported defect types should include:
- `orphan_workflow`
- `missing_target_workflow`
- `unparseable_target_workflow`
- `ambiguous_target_reference`
- `decomposed_unwired_workflow`
- `reframework_wiring_inconsistency`
- `root_entrypoint_missing`
- `graph_discontinuity`

Severity should include at least:
- `execution_blocking`
- `handoff_required`

Examples:
- missing Main.xaml or missing invoked target is `execution_blocking`
- decomposed helper workflow emitted but unused is usually `handoff_required`
- ambiguous target that prevents deterministic traversal is `execution_blocking`

6. Add regression tests
Create `server/__tests__/workflow-graph-validator.test.ts`.

Test cases should include:
- clean package where Main.xaml invokes a valid chain of workflows and all are reachable
- orphaned executable workflow present in the package but never reachable from Main.xaml
- broken InvokeWorkflowFile reference to a missing file
- target file exists but contains invalid/unparseable XAML so traversal fails
- REFramework-like package missing one expected state workflow
- dispatcher/performer package where decomposed workflow exists but is never wired into execution
- ambiguous reference case where more than one similarly named workflow could match
- package with non-executable support files that should not be flagged as orphans
- clean package passes with `hasWorkflowGraphIntegrityIssues: false`

Verify:
- correct defect classification
- correct severity assignment
- proper root-based reachability calculation
- no false positives on excluded non-executable files
- final structured diagnostics contain all required fields

7. Keep diagnostics visible to downstream consumers
Ensure the resulting graph diagnostics can flow into:
- final quality report
- DHG generation
- debug/admin surfaces if already supported by the reporting model

The graph validator should produce structured output that downstream consumers can summarize without re-parsing XAML.
