# CB2-F22A - Contract Extraction Hardening and Non-Contract Property Exclusion

What & Why
The first implementation of workflow contract integrity validation is useful, but the latest package run shows that it is still overmatching non-contract properties and designer metadata as if they were real child-workflow arguments. This creates false-positive contract defects such as unknown target arguments for fields that are not actual invoke bindings. At the same time, real contract issues still remain: undeclared symbols, mixed literal/expression syntax, invalid invoke serialization, and placeholders in required executable properties.

This task hardens the contract extractor so it only validates true workflow contract surfaces and explicitly excludes designer metadata and non-contract properties. The goal is to reduce false positives without hiding real contract defects.

Goals
1. Validate only real parent/child workflow contract bindings.
2. Exclude designer metadata and non-contract properties from invoke argument analysis.
3. Preserve and sharpen real defects around undeclared symbols, bad invoke serialization, and mixed literal/expression syntax.
4. Improve defect precision so DHG and quality reporting are more trustworthy.

Done looks like
A hardened contract extraction layer distinguishes:
- actual workflow target file references
- actual bound invoke arguments
- designer metadata
- view-state/layout fields
- non-contract activity properties

The validator no longer reports designer metadata or non-contract properties as `unknown_target_argument` or `invoke_argument_binding_mismatch`.

The validator still correctly reports:
- undeclared_variable_reference
- undeclared_argument_reference
- invalid_expression_scope
- unknown_target_argument for true argument bindings only
- missing_required_target_argument
- invoke_argument_binding_mismatch
- invalid_invoke_serialization
- mixed_literal_expression_syntax
- required_property_placeholder
- decomposed_workflow_missing_contract

A structured exclusion/ignore report is available for debugging, so it is clear which fields were intentionally ignored as non-contract/designer metadata.

The final quality report becomes more precise:
- fewer false-positive contract defects
- unchanged or clearer real contract defects
- no silent suppression of genuine issues

Build mode is preserved. Package and DHG still generate.

Key invariants
- Pattern-first, not activity-first.
- Only true workflow contract carriers should participate in contract matching.
- Designer metadata, annotations, view-state, layout hints, and similar non-contract properties must never be treated as child workflow arguments.
- Do not reduce defect count by hiding genuine contract problems.
- Do not invent symbols, arguments, or values.
- Do not hardcode legacy status strings.

Out of scope
- Reworking the whole decomposition model
- Inventing missing business logic
- Replacing existing executable-path, graph, or dependency validators
- Broad activity-specific hacks except for a minimal exclusion allowlist where genuinely required by UiPath serialization behavior

Tasks

1. Harden contract extraction logic
Update `server/xaml/workflow-contract-integrity.ts` so contract extraction explicitly separates:
- workflow target resolution
- true invocation argument bindings
- designer metadata
- view-state/layout fields
- other non-contract properties

Only true invocation argument bindings should be compared against child workflow declared arguments.

2. Add explicit non-contract exclusion rules
Create a small, explicit exclusion layer for fields that must never be treated as contract arguments, including categories such as:
- designer metadata
- view-state fields
- layout sizing/position hints
- IdRef-like references
- annotation-only fields
- other non-runtime/non-contract serialization artifacts

Do not just drop them silently. Record them in a structured informational bucket such as:
- `contractExtractionExclusions[]`
with fields like:
- file
- workflow
- activityType
- propertyName
- exclusionReason

3. Tighten invoke binding parsing
Make invoke binding parsing recognize only supported contract-bearing forms, such as:
- canonical argument dictionaries/maps
- explicit bound argument entries
- equivalent supported serialized forms used by the pipeline

It must not treat every property present on an invoke activity as a candidate child argument.

4. Preserve real invalid invoke serialization defects
If an invoke activity contains pseudo-properties or conflicting serialization shapes, do not misclassify them as unknown child arguments by default. Instead:
- classify them under `invalid_invoke_serialization`
- preserve the raw offending property/value in diagnostics
- keep them eligible for downgrade

5. Refine defect classification priority
When both of these are true:
- the field is not a valid contract carrier
- the serialization is malformed

Prefer classification as:
- `invalid_invoke_serialization`
rather than:
- `unknown_target_argument`

Reserve `unknown_target_argument` for true bound arguments that target a child workflow argument name which does not exist.

6. Add summary metrics
Extend `contractIntegritySummary` with metrics such as:
- totalContractDefects
- totalNormalizationActions
- totalExecutionBlocking
- totalHandoffRequired
- totalUnknownTargetArguments
- totalInvalidInvokeSerialization
- totalUndeclaredSymbols
- totalExcludedNonContractFields

7. Integrate into reporting
Ensure FinalQualityReport includes:
- `contractIntegrityDefects`
- `hasContractIntegrityIssues`
- `contractIntegritySummary`
- `contractNormalizationActions`
- `contractExtractionExclusions`

If exclusions are present, expose them in debug/reporting surfaces, but do not treat them as defects.

8. Add regression tests
Create or extend:
`server/__tests__/workflow-contract-integrity.test.ts`

Add cases for:
- designer metadata field on an invoke activity is excluded, not treated as child argument
- view-state/layout property is excluded, not treated as child argument
- true unknown child argument still triggers `unknown_target_argument`
- malformed invoke serialization triggers `invalid_invoke_serialization`
- conflicting argument blob plus pseudo-properties is classified correctly
- undeclared variable and undeclared argument defects still fire normally
- summary counts reflect exclusions separately from defects

Verify:
- false-positive contract defects decrease
- real contract defects remain visible
- exclusions never silently erase genuine issues
- no fabricated symbols or bindings are introduced
