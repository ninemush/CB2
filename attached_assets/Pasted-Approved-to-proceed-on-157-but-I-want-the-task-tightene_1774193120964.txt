Approved to proceed on #157, but I want the task tightened so it fixes the regression without undermining the architecture introduced in #156.

Important:
This is a regression fix.
I do NOT want a rollback of the OIDC-driven architecture from #156.
I want a precise correction to the bad scope-request policy that was introduced.

Root problem
The bug is not OIDC itself.
The bug is that the runtime started requesting only minimal scope subsets for operational tokens.

For UiPath, the token only contains the scopes that are requested.
So if CannonBall requests only 2-3 OR scopes, the resulting JWT is missing the scopes needed for folders, robots, machines, queues, assets, etc., and those calls correctly fail with 403.

What must be true after this fix
- OIDC remains the source of truth for which scopes are valid
- saved/configured connection scopes remain the source of truth for what CannonBall requests for the user’s operational tokens
- minimal scope selection must never be used for the main operational OR/service tokens that power the app
- minimal scope logic, if retained at all, must be restricted to explicitly bounded diagnostic scenarios only

Please revise and execute #157 with these constraints

1. Scope request policy
Separate these two concepts explicitly:

A. Valid scopes
- derived from OIDC discovery
- used to know which scopes/families are real

B. Requested scopes for token acquisition
- for real operational tokens, request the full relevant configured/saved scope set
- do NOT collapse the requested scope set to a minimal subset for operational use

For OR specifically:
- if the user has saved scopes in the DB/config, request those
- do not replace them with a minimal OR.Default / OR.Administration / OR.Execution subset

For non-OR services:
- request the full relevant OIDC-backed family scopes or the configured/supported set for that service
- do not reduce them to a single minimal read/default scope when acquiring the token used by real app flows

2. Respect persisted scope configuration
Fix loadConfig() so it actually reads and preserves the saved scopes from:
- uipath_connections.scopes
- appSettings.uipath_scopes where relevant

Only fall back to default/minimal scopes when no saved/configured scopes exist at all.

3. Minimal scopes must be bounded
If pickMinimalFromOidc() or equivalent remains in the codebase:
- rename or document it so it is clearly diagnostic-only
- ensure it is NOT used by:
  - connection test
  - folder loading
  - service probes
  - deployment
  - live operations
  - scope verification intended to reflect real granted scopes

4. Auto-detect / scope verification behavior
The auto-detect and verification flow should request the full relevant scope families so UiPath can grant everything the External App is actually allowed to receive.

I want the resulting “Granted Scopes” view to reflect reality, not an artificially minimal request.

5. No regression to the #156 architecture
Do not undo:
- OIDC as validity truth
- MetadataService-based scope authority
- snapshot fallback chain
- per-service degradation model

This task should correct the operational token request logic, not dismantle the architecture.

6. Regression protection
Add tests that would have caught this exact bug.

At minimum verify:
- a saved connection with broad OR scopes loads those scopes correctly
- operational OR token requests include the full configured/saved OR scope set
- granted scopes are not collapsed to 3 OR scopes when broader scopes are saved/configured
- folder loading succeeds again
- connection test succeeds again
- Orchestrator probe succeeds again
- previously working services remain working

7. Final report
After implementation, report back with:

- exact root cause confirmed in code
- what changed in loadConfig()
- what changed in MetadataService scope resolution
- what changed in token acquisition logic
- whether minimal-scope logic still exists and where it is allowed to be used
- which operational flows were verified restored:
  - connection test
  - folder loading
  - service probe
  - live operations
  - deployment-related OR access
- what regression tests now protect this path

Acceptance criteria
- OIDC remains the source of truth for valid scopes
- operational tokens request the full relevant configured/saved scope sets
- saved user scopes are no longer discarded
- minimal scope logic is not used for operational tokens
- Orchestrator connectivity is restored
- folder loading works again
- granted scopes reflect the real granted set, not an artificially minimized request
- previously working services remain working
- no rollback of the broader #156 architecture
