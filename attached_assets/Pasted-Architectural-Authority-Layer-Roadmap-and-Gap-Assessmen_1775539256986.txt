Architectural Authority-Layer Roadmap and Gap Assessment

What & Why
The current pipeline still behaves too much like:
generate pseudo-automation -> emit XML -> run layered corrections -> inspect failures -> patch symptoms

That is not how a strong UiPath Solution Architect or senior developer builds in Studio.

A senior developer effectively works with a small set of authoritative truths before runtime artifacts exist:
- symbol table
- workflow contracts
- activity schema and required properties
- expression validity
- graph connectivity
- dependency closure

The repeated defect pattern across many runs suggests CB2 still lacks a few of these authority layers, or applies them too late. As a result, truth is still being discovered at validation time instead of being known during assembly.

This task is not about fixing one defect family from one run. It is about using the current codebase and known benchmark evidence to identify the remaining missing or non-authoritative layers, rank them by leverage, and define the fewest next engineering tasks required to move CB2 closer to Studio-like construction.

Goal
Produce a concrete authority-layer gap assessment and implementation roadmap that answers:
1. Which authority layers already exist and are reasonably strong
2. Which authority layers are incomplete, duplicated, or too late in the pipeline
3. Which 2 to 4 remaining engineering tasks will most directly move CB2 toward Studio-stable output with business logic preserved and an honest DHG

Done looks like
- The codebase is assessed against a small set of authority layers rather than individual run symptoms.
- Each authority layer is classified as:
  - strong
  - partial
  - weak/missing
- For each weak or partial layer, the assessment identifies:
  - where it currently exists in code
  - where it is bypassed or duplicated
  - whether truth is established early enough
  - what the highest-leverage fix would be
- The output includes a ranked roadmap of the next 2 to 4 tasks, each tied to an authority gap rather than a one-off symptom.
- The roadmap explicitly aims to reduce dependence on post-emission repair and late-stage discovery.

Authority layers to assess
Use this set unless the code proves a better grouping:
1. Symbol authority
   - variables and arguments discovered, typed enough, scoped, declared before quality gate
2. Workflow contract authority
   - caller/callee argument contracts known before emission
3. Activity schema authority
   - namespace, required properties, child-element vs attribute shape, package ownership
4. Expression authority
   - no JSON intermediates or malformed executable values reach XML serialization
5. Graph authority
   - required workflows are reachable from entrypoint before finalization
6. Dependency authority
   - final emitted activity/workflow truth determines package dependencies proactively
7. Reporting authority
   - DHG and diagnostics reflect the real final state honestly and consistently

Tasks
1. Review the current codebase against the authority layers above.
   For each layer, identify:
   - primary implementation files
   - duplicate or competing paths
   - whether it acts before or after emission
   - whether it is authoritative or only advisory/repair-based

2. Use the benchmark evidence already available from BirthdayGreetings and Coupa runs as validation points, but do not build the assessment around one run only.
   The goal is architectural diagnosis, not another run-by-run defect summary.

3. Produce an authority-layer gap table with, for each layer:
   - current status: strong / partial / weak
   - key code paths
   - common failure manifestations
   - likely reason truth is established too late or inconsistently
   - recommended fix direction

4. Identify the minimum next task set.
   Recommend the next 2 to 4 tasks only.
   Each task must:
   - target an authority gap
   - have clear blast radius
   - avoid broad speculative rewrites
   - avoid symptom-only regex repair where a stronger authority fix exists

5. Explicitly call out what should NOT be done next.
   Examples:
   - broad combined mega-task
   - new parallel validator if one already exists
   - another post-emission patch when an upstream authority fix is needed

6. Keep the output practical.
   This should end with a ranked roadmap that can be turned directly into the next Replit tasks.

Out of scope
- Implementing the fixes in this task
- Another run-specific forensic report
- Prompt engineering as the main solution
- Cosmetic UI/reporting suggestions not tied to an authority gap
