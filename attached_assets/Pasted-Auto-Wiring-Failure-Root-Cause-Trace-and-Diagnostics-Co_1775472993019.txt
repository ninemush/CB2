Auto-Wiring Failure Root-Cause Trace and Diagnostics Consolidation

What & Why
Recent runs across multiple use cases show the same end result:
- workflows are generated from spec decomposition
- but some required workflows remain generated-but-unwired or unreachable from Main.xaml
- the package is downgraded to structurally invalid / handoff-only

However, the codebase already appears to contain substantial auto-wiring, reachability, and required-workflow enforcement logic. So the likely gap is not “missing auto-wiring architecture.” The likely gap is:
- a bug in the existing auto-wiring/classification path
- and a lack of one consolidated diagnostic artifact explaining why specific workflows were not wired

This task must not build a new graph/wiring framework. It must investigate and instrument the existing live auto-wiring path, consolidate diagnostics, and expose the exact reason each generated-but-unwired or unreachable required workflow failed to wire.

Goals
1. Trace the existing active-path auto-wiring logic end to end.
2. For every generated-but-unwired or unreachable required workflow, identify the exact failure reason.
3. Emit one structured diagnostics artifact that explains wiring outcomes clearly.
4. Patch only the specific failing branch(es) once the trace proves the root cause.

Done looks like
- The pipeline emits a structured auto-wiring diagnostics artifact for every generated workflow.
- For each required workflow, the artifact shows:
  - whether it was expected to be wired
  - expected caller
  - whether invoke emission was attempted
  - whether invoke emission succeeded
  - whether the final emitted graph made it reachable
  - exact failure reason if not wired or unreachable
- Generated-but-unwired and unreachable required workflows are no longer opaque late warnings. They have explicit root-cause diagnostics.
- The diagnostics use the actual finalized workflow set consumed by package assembly and reachability validation, not stale intermediate data.
- If one or more concrete failing branches are identified in the current logic, patch those branches without introducing a parallel graph framework.

Out of scope
- Rebuilding the workflow graph / auto-wiring system from scratch
- Broad dependency-family redesign
- New family-lowering abstractions
- XAML namespace/prefix fixes unrelated to wiring diagnostics
- UI-only reporting changes without root-cause traceability

Tasks

1. Instrument existing required-workflow classification
Trace how the current code classifies workflows as required vs optional/helper.
For each generated workflow, record:
- workflow/file
- generated: true/false
- required: true/false
- classification source / rule
- any severity mapping that makes it package-fatal if unwired

2. Instrument existing caller/callee auto-wiring path
At the current live auto-wiring path, record for each workflow:
- expected caller
- actual caller candidates considered
- whether invoke emission was attempted
- whether invoke emission succeeded
- if skipped, why it was skipped
- if rejected, why it was rejected
- argument contract / signature reason if that was the blocker

3. Instrument reachability on the finalized graph
Run reachability against the actual finalized workflow/XAML set used for package assembly.
For each required workflow, record:
- reachableFromMain: true/false
- actualCallers discovered from emitted graph
- missing edge if unreachable
- whether the graph analyzer used final or stale workflow content

4. Emit consolidated diagnostics artifact
Add:
workflowAutoWiringDiagnostics

Per workflow include:
- file
- generated
- required
- expectedCaller
- callerCandidates
- invokeAttempted
- invokeSucceeded
- invokeSkippedReason
- invokeRejectedReason
- actualCallers
- reachableFromMain
- finalWiringStatus
- rootCauseCategory
- remediationHint

Summary counts:
- totalGeneratedWorkflows
- totalRequiredWorkflows
- totalRequiredWorkflowsWired
- totalGeneratedButUnwired
- totalUnreachableFromMain
- totalMissingInvokeEdges
- totalCallerMismatchDefects
- totalInvokeEmissionFailures
- totalStaleGraphAnalysisViolations

5. Add active-path proof
Emit a lightweight trace showing that:
- the same finalized workflow set used by package assembly
- is the workflow set used by auto-wiring and reachability diagnostics

For each run record:
- finalizedWorkflowSetHash
- autoWiringDiagnosticsWorkflowSetHash
- reachabilityWorkflowSetHash
- hashesMatch: true/false

In debug/test mode, fail if diagnostics are computed on stale or pre-finalized workflow data.

6. Investigate and patch actual failing branches
Use the new trace on the known failing benchmark cases:
- BirthdayGreetingsV26
- CoupaInvoiceReconciliation

If the trace shows a concrete bug in:
- caller selection
- invoke emission
- classification
- reachability using stale data
patch that specific branch.
Do not introduce a parallel auto-wiring framework.

7. Add focused regression tests
Cover:
- required generated workflow gets wired and becomes reachable
- required generated workflow is not wired and diagnostics show exact cause
- optional/helper workflow can remain unwired without being classified as package-fatal
- debug/test mode fails if diagnostics use stale workflow data
- benchmark-style case where invoke emission is skipped now produces explicit rootCauseCategory
