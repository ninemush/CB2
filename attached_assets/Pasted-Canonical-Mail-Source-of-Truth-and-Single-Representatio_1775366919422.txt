Canonical Mail Source-of-Truth and Single Representation Enforcement

What & Why
The latest runs show the same dominant mail failure pattern:
- Send SMTP Mail Message emitted with missing required Body
- Send Gmail Message emitted with missing required Body
- UiPath.Mail.Activities still appearing or aligning downstream even when the business intent is connector-driven Gmail send

The structured spec currently carries competing representations of the same logical mail operation:
- a narrative TryCatch that says to send via Gmail Integration Service connector using InGmailConnectionName
- a separate concrete GmailSendMessage step using UiPath.GSuite.Activities with To, Subject, and Body=summaryBody

This is the real drift point. One logical send operation is surviving downstream in more than one representation, and later stages are not choosing one canonical executable model early enough.

The codebase today does not have a first-class executable "Integration Service mail" family wired through lowering, and UiPath.IntegrationService.Activities entries are not emission-approved. So the fix should not invent a fourth executable family without a concrete target. Instead:
- preserve connector intent as the source-of-truth signal
- choose one concrete, emission-approved executable mail target for that cluster
- do not allow multiple competing mail representations to survive past lowering

The likely valid concrete targets are the existing mail families already supported by lowering, with preference rules based on connector intent and active Studio target/profile. If connector-driven mail intent is present, the lowering step must select one valid concrete executable representation and reject all others.

Goals
1. Treat connector-driven mail intent as the source-of-truth signal for a logical mail cluster.
2. Ensure each logical mail cluster has exactly one canonical executable representation before XAML emission.
3. Eliminate competing representations such as:
   - narrative TryCatch connector send
   - concrete GmailSendMessage
   - generic Mail/SMTP fallback
   surviving in the same cluster.
4. Preserve To, Subject, Body, and connection-related context through canonical lowering.
5. Fail early if no valid concrete executable representation can be chosen for the active target/profile.

Done looks like
- Each logical mail cluster is identified before XAML emission.
- A cluster has exactly one canonical executable representation.
- If connector-driven mail intent is present, it influences family selection, but only one emission-approved concrete target is allowed to survive.
- Narrative TryCatch mail descriptions are either:
  - lowered into real TryCatch containers around the chosen concrete mail activity
  - or rejected before emission
- Once a cluster is resolved to a concrete target, no wrong-family artifacts may appear later for that cluster.
- If Body exists in the source cluster, it must be preserved. If it is lost during lowering, lowering fails before emission.
- mailSourceOfTruthDiagnostics is emitted with cluster-level selection and rejection detail.
- No placeholder injection for rejected mail clusters.

Out of scope
- Broad catalog/filter alignment
- Adding new IntegrationService executable activity support if no valid concrete target exists
- General parity/freeze changes
- Full decomposition redesign
- Non-mail cluster families

Tasks

1. Add logical mail-cluster detection
Group related nodes representing one logical send operation, including:
- concrete send steps
- adjacent retry wrappers
- adjacent TryCatch wrappers
- nearby catch/logging steps tied to the same send intent

2. Add canonical source-of-truth selection
For each cluster, detect all present mail representations:
- connector-driven narrative intent
- GSuite/Gmail concrete activity
- generic Mail/SMTP concrete activity
- narrative TryCatch container text
Choose one canonical source-of-truth for the cluster.
If connector-driven intent is present, it must influence selection, but only via a valid concrete executable mapping for the active target/profile.

3. Add concrete target selection
Map the chosen cluster intent to exactly one concrete, emission-approved executable target valid for the current Studio profile and verified dependency set.
Do not leave the cluster at an abstract "connector" family.
If no valid concrete target exists, fail before emission with structured diagnostics.

4. Eliminate competing representations
If a cluster contains:
- narrative TryCatch connector send
- concrete GmailSendMessage
- generic SMTP/Mail activity
for the same logical operation,
do not emit all of them.
Either:
- lower them into one canonical executable structure
- or reject the cluster before emission

5. Preserve required properties through lowering
For the chosen target, preserve:
- To
- Subject
- Body
- connector or account context if applicable
If Body exists upstream but is lost during lowering, fail before emission.
Do not allow placeholder injection for rejected clusters.

6. Add downstream drift guardrails
Once a cluster is locked to its chosen concrete target:
- no wrong-family activity tags may appear later
- no wrong-family package may appear later
- no fallback artifacts may appear later
If they do, treat it as package-fatal source-drift, not a late warning.

7. Emit diagnostics
Add:
mailSourceOfTruthDiagnostics
with per cluster:
- workflow/file
- clusterId
- detectedRepresentations
- selectedCanonicalSource
- selectedConcreteTarget
- selectedPackage
- target/profile compatibility
- requiredPropertiesSeen
- missingRequiredProperties
- rejectedCompetingRepresentations
- packageFatal
- remediationHint

Summary counts:
- totalMailClusters
- totalResolvedToSingleRepresentation
- totalRejectedForCompetingRepresentations
- totalRejectedForNoConcreteTarget
- totalBodyLossDuringLowering
- totalWrongFamilyDriftBlocked

8. Integrate with package completeness policy
Rejected or incomplete mail clusters must flow into packageCompletenessViolations and fail package mode early and honestly.

9. Regression tests
Cover:
- connector-driven mail cluster resolves to one concrete executable target only
- cluster with narrative TryCatch plus Gmail concrete step does not emit both
- Gmail-intent cluster does not drift into SMTP/Mail artifacts unless explicitly selected
- Body present upstream is preserved
- wrong-family package appearance after lock is package-fatal
- no placeholder injection for rejected clusters
