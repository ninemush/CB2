#384 - Add representative package-generation regression coverage and source-of-truth validation

What & Why
The latest review shows that several previously suspected issues are already merged or are lower severity than originally feared. That is good news.

What is still missing is confidence that the package-generation module works correctly as a single unit for the representative use-case families we actually care about.

Right now we still do not have regression coverage for the real mixed-system scenarios that have been failing in practice:
- Birthday Greetings
- Travel
- Password Reset

Without those fixtures, we keep proving helper behavior and isolated fixes, but not the actual end-to-end consistency of:
- activity-family selection
- package/prefix/namespace/declaration truth
- honest degradation for unsupported families
- smoke-test survival
- DHG truthfulness

This task is not a broad refactor. It is a focused hardening pass that validates current logic against representative scenarios and official first-party UiPath sources, then fixes the concrete gaps those scenarios expose.

Done looks like
The repo contains regression fixtures/tests for the representative use-case families:
- Birthday Greetings family
- Travel family
- Password Reset family

Those tests validate the package-generation module as a unit, including:
- selected activity families
- package and prefix mapping
- declaration/header correctness
- smoke-test compatibility
- honest degradation where a family is still not emission-approved
- DHG consistency with emitted generated vs degraded output

The remaining silent catalog-load catch blocks in workflow-tree-assembler.ts are replaced with visible warning logging.

The dead/inconsistent GSuite prefix mapping is cleaned up so registry/compliance naming is consistent and does not leave misleading unused mappings.

Official first-party UiPath documentation is used to validate the representative package families already referenced by these use cases, especially:
- UiPath.GSuite.Activities
- UiPath.Mail.Activities
- UiPath.WebAPI.Activities
- UiPath.Persistence.Activities
- UiPath.DataService.Activities
- UiPath.CommunicationsMining.Activities

Any concrete mismatch discovered between code and official package/activity metadata during this audit is corrected in the same task.

Out of scope
Large architecture refactors
Changing AI prompts
Weakening the smoke test
Weakening dependency validation
Adding broad new activity-family support beyond what is already referenced in these use cases
Speculative DHG/package-contract changes without a demonstrated failing case

Tasks
1. Add representative regression fixtures/tests for the three analyzed use-case families.
At minimum cover:
- Birthday Greetings family: Google Calendar + Google Contacts + Gmail + queue/orchestrator + honest degradation for still-unapproved Google/GenAI families where applicable
- Travel family: API/Web or connector-style data collection + Data Service + Gmail/mail + Action Center/human-review path
- Password Reset family: email intake + admin-portal/UI path + Action Center/manual-review path + Communications Mining honest degradation if still unsupported

2. Make the regression tests validate the package-generation pipeline as a unit, not just helpers.
The assertions should cover:
- activity-family selected for each representative step
- package/prefix/namespace consistency
- declaration/header generation for emitted activities
- smoke-test compatibility for supported families
- honest degradation instead of half-generation for unsupported families
- DHG not overstating generated coverage for degraded paths

3. Replace the remaining bare catalogService.load() catch blocks in workflow-tree-assembler.ts with visible warning logging.
Do not change behavior to a hard failure; just make degraded/no-catalog mode visible.

4. Clean up the dead/inconsistent GSuite prefix mapping.
Make registry/compliance naming consistent for UiPath.GSuite.Activities so there is one canonical prefix path and no misleading unused reverse mapping.

5. Validate the representative supported package families against official first-party UiPath docs.
Use official UiPath documentation to confirm package ids, namespaces/prefix expectations, and family intent for:
- GSuite / Google Workspace
- Mail
- WebAPI
- Persistence / Action Center
- Data Service
- Communications Mining
If the audit finds a real mismatch in current code, fix it in the same task.

6. Only if the new representative regressions expose a concrete DHG truthfulness bug, fix that bug in the same task.
Do not broaden scope into speculative DHG refactoring without a demonstrated failing example.
