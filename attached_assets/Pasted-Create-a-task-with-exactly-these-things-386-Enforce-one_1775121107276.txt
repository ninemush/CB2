#386 - Enforce one authoritative XAML normalization and package-contract boundary

What & Why
We have been attempting upstream/systemic fixes, but the current repo still shows a gap between intended architecture and actually enforced invariants.

The package-generation module still allows different layers to serialize values, types, prefixes, and declaration structures differently. That is why the pipeline can contain partial fixes and still produce Studio-breaking output.

This task is not a new philosophy. It is the implementation-completion task for the philosophy we already chose:
- Studio-openable package first
- honest DHG
- safe degradation
- no fake completeness

The goal is to make the package pipeline authoritative at one normalization boundary so no later layer can reinterpret or corrupt what an earlier layer meant.

Done looks like
All emitted XAML passes through one canonical normalization boundary for:
- property values
- typed property objects
- variable defaults
- literal vs expression classification
- variable and argument types
- generic/nested type arguments
- package/prefix/xmlns/assembly/import mappings

The Emission Gate cannot mutate declaration infrastructure or declaration-section contents.

The package-generation pipeline uses one canonical package-family mapping source for:
- package id
- canonical prefix
- xmlns declaration
- clr namespace
- assembly reference

Representative regressions prove the pipeline as a unit for:
- Birthday Greetings family
- Travel family
- Password Reset family

No package is emitted if any of these invariant classes remain violated:
- raw typed property objects leaked into XAML
- raw CLR type names leaked where XAML-prefix types are required
- unwrapped VB expressions in literal/default slots
- broken generic type serialization
- declaration infrastructure corruption
- package/prefix/declaration mismatch
- DHG generated-vs-degraded mismatch

Out of scope
Adding new activity families beyond the currently analyzed use-case families
Changing AI prompts
Weakening the smoke test
Weakening dependency validation
Broad architecture rewrites outside the package-generation boundary

Tasks
1. Create one canonical normalization layer in workflow-tree-assembler.ts (or a dedicated shared module) and route all XAML emission paths through it.
This layer must own:
- property value normalization
- typed property object resolution
- literal vs VB-expression classification
- variable default normalization
- variable/argument type normalization
- generic/nested type serialization

2. Remove duplicate ad hoc serialization behavior.
Audit current code paths that still directly use String(...), smartBracketWrap(...), normalizeStringToExpression(...), or custom wrapping logic and route them through the canonical normalization layer instead of allowing per-path behavior differences.

3. Enforce one canonical package/prefix/declaration source of truth.
Unify the mapping used across:
- uipath-activity-registry.ts
- xaml/xaml-compliance.ts
- catalog-driven XAML emission
There must be one canonical mapping for each supported package family:
- package id
- prefix
- xmlns
- clr namespace
- assembly
Clean up dead or misleading alternate mappings such as the current GSuite prefix inconsistency.

4. Make source-of-truth degradation visible.
Replace remaining silent catalogService.load() catch blocks in workflow-tree-assembler.ts with visible warning logging. Do not convert to hard failure, but do not allow silent degraded/no-catalog mode.

5. Protect declaration infrastructure as an invariant.
Ensure the Emission Gate and any downstream mutation pass cannot modify:
- AssemblyReference
- Collection / sco:Collection
- TextExpression.ReferencesForImplementation contents
- TextExpression.NamespacesForImplementation contents
Add regression tests that prove these survive intact.

6. Add artifact-verified regression coverage for the three representative use-case families.
The tests must validate the package-generation pipeline as a unit, including:
- selected activity families
- package/prefix/xmlns/assembly/import consistency
- declaration/header correctness
- smoke-test compatibility for supported families
- honest degradation for unsupported families
- DHG truthfulness for generated vs degraded output

7. Add compiler-invariant regression checks for the defect patterns already observed.
At minimum lock down:
- no typed property objects in emitted XAML
- no raw CLR type leakage where XAML-prefix types are required
- no unwrapped VB expression defaults
- no malformed generic argument serialization
- no mixed literal/expression attribute syntax
- no declaration-block corruption

8. Add one final pre-emit invariant gate.
Before a package is emitted, fail if any canonical normalization invariant remains violated. Do not rely on later layers to repair malformed upstream output.

Important constraints
Do not weaken the smoke test.
Do not weaken dependency validation.
Do not solve this by adding more warnings.
Do not leave multiple serialization rules for the same kind of value/type across different layers.
Do not let later layers reinterpret malformed upstream output.
The purpose of this task is to make the package-generation module authoritative as a single unit, not a series of inconsistent partial filters.
