Critical Workflow Auto-Wiring and Reachability Enforcement

What & Why
After the recent fixes to quote safety, ValueIntent serialization, invoke dotted-tag prefix consistency, and mail Body child-element handling, the dominant shared blocker across multiple use cases is now workflow graph assembly.

Recent runs show the same pattern across at least two different automations:
- workflows are successfully generated from spec decomposition
- but they are not auto-wired into the orchestration graph
- multiple generated workflows remain unreachable from Main.xaml or the process entry path
- packages end up structurally invalid or handoff-only even though useful workflow pieces exist

This is now the main platform-level bottleneck. The issue is no longer only per-workflow XAML correctness. It is end-to-end graph assembly correctness.

The fix must make required workflow reachability a first-class invariant:
- if a workflow is required by the declared process, it must be wired into the orchestration graph
- if it cannot be wired deterministically, the pipeline must fail with explicit graph-assembly diagnostics rather than leaving generated but unreachable workflows as late warnings

Goals
1. Ensure all required generated workflows are either:
   - deterministically wired into the executable graph
   - or explicitly failed as graph-assembly defects before package completion
2. Eliminate "generated but not wired" and "unreachable from Main.xaml" as late-stage surprises for required workflows.
3. Make workflow reachability and caller/callee linkage explicit and auditable.
4. Improve cross-use-case robustness by treating graph assembly as a platform capability, not a use-case-specific patch.

Done looks like
- Every workflow generated from spec decomposition that is required by the declared process has an identified caller and an emitted invoke path from the appropriate entry workflow.
- Generated-but-unwired required workflows no longer survive as late warnings. They become graph-assembly defects earlier if they cannot be wired.
- Unreachable required workflows from Main.xaml or other declared entry paths are reduced to zero for the BirthdayGreetingsV26 and CoupaInvoiceReconciliation benchmark cases.
- A structured graph wiring diagnostics artifact explains:
  - which workflows were generated
  - which workflows were classified as required vs optional
  - which caller was expected to invoke each required workflow
  - whether the invoke node was emitted
  - why any workflow remained unreachable
- Existing validators still run, but graph assembly defects are surfaced earlier and more explicitly.

Out of scope
- Fixing per-workflow XAML correctness unrelated to reachability
- Broad dependency-family mapping redesign
- Mail Body child-element behavior beyond what already exists
- New family-selection abstractions
- UI-only reporting changes

Tasks

1. Define required-workflow classification
Add a deterministic classification step that marks each generated workflow as:
- required
- optional/helper
- unreachable-by-design only if explicitly justified
This classification must be derived from the declared process structure, not guessed from filenames alone.

2. Add explicit caller/callee wiring plan
Before final XAML emission completes, build a wiring plan that records for each required workflow:
- expected caller workflow
- expected invoke activity type/form
- expected argument contract reference
- whether the invoke node was actually emitted

3. Enforce auto-wiring for required workflows
If a workflow is classified as required and no valid invoke path is emitted from its expected caller chain, do not let it survive as only a late warning.
Record a graph-assembly defect with explicit reason and fail package mode / downgrade appropriately.

4. Add reachability verification against the actual emitted graph
Run a reachability pass on the final emitted workflow graph:
- start from Main.xaml or other declared entry workflow
- traverse actual invoke relationships
- confirm every required workflow is reachable
If not reachable, record the exact missing edge or caller mismatch.

5. Emit graph assembly diagnostics
Add:
workflowGraphAssemblyDiagnostics

Per workflow include:
- file
- workflowRole
- required: true/false
- expectedCaller
- actualCallers
- invokeEmitted: true/false
- reachableFromMain: true/false
- graphAssemblyStatus
- defectReason
- remediationHint

Summary counts:
- totalGeneratedWorkflows
- totalRequiredWorkflows
- totalRequiredWorkflowsWired
- totalGeneratedButUnwired
- totalUnreachableFromMain
- totalMissingInvokeEdges
- totalCallerMismatchDefects

6. Add active-path proof
Emit a trace artifact proving that the same final XAML entries used for package assembly were also used for graph reachability evaluation.
In debug/test mode, fail if graph analysis runs on a stale or pre-finalized workflow set.

7. Tighten package completeness integration
Generated-but-unwired required workflows and unreachable required workflows must be treated as package-fatal graph defects, not only post-assembly warnings.

8. Add regression tests
Cover:
- required child workflow is deterministically wired and reachable
- generated workflow with no emitted invoke edge becomes a graph-assembly defect
- optional/helper workflow can remain unwired without failing required-reachability rules if explicitly classified as optional
- Main plus multiple child workflows all reachable after emission
- debug/test mode fails if graph analysis sees stale workflow data
