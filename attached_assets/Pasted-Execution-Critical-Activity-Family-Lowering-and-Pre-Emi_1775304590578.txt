Execution-Critical Activity Family Lowering and Pre-Emission Contract Synthesis

What & Why
The latest runs show that package integrity checks are now stricter and more honest, but the generator still emits structurally incomplete execution-critical activities. The clearest recurring failure is mail sending:

- final packages repeatedly contain Send Gmail Message and Send SMTP Mail Message activities missing required Body properties
- package mode previously tried to heal this with PLACEHOLDER injection
- the structured workflow spec already contains enough intent in many cases (for example, summaryBody and summarySubject variables are explicitly built, and a GmailSendMessage step with To/Subject/Body is present in the spec), yet the final XAML still ends up with incomplete send activities

This means the problem is now upstream in the lowering/compiler path from structured workflow spec to concrete UiPath activity instances.

The system currently allows ambiguous or mixed representations such as:
- pseudo-steps described as Try/Catch literals instead of executable child activities
- multiple candidate mail activity families being inferred or mixed
- activity-family selection happening too late or inconsistently
- required runtime contracts being checked after emission instead of before emission

This task introduces family-aware lowering contracts for execution-critical activity groups and enforces completeness before final XAML emission.

Goals
1. Convert execution-critical structured steps into one concrete, valid activity-family representation before XAML emission.
2. Enforce required property completeness for each critical activity family before emission.
3. Prevent mixed or fallback activity-family drift (for example Gmail intent turning into incomplete SMTP artifacts).
4. Fail early with structured diagnostics when a critical activity contract cannot be satisfied.
5. Reduce dependence on post-assembly healing for critical send/task/update operations.

Done looks like
- Each execution-critical step is lowered into exactly one concrete activity family before final XAML emission.
- Mail-send intent does not produce mixed Gmail/SMTP fallback artifacts unless explicitly designed and fully contracted.
- Required properties for execution-critical activity families are satisfied before XAML emission or the workflow is rejected as package-fatal/handoff-only.
- Pseudo-activity narrative representations (for example Try/Catches written as plain-text property payloads) are either lowered into real child activities/containers or rejected before emission.
- New diagnostics clearly show which critical activity family was selected, what contract was required, whether it was satisfied, and why lowering failed if it failed.
- Post-assembly placeholder injection for critical activities is no longer needed in normal operation.

Out of scope
- Broad catalog/filter alignment
- New parity/freeze concepts
- Reworking the full decomposition strategy
- Deep business-logic invention
- UI-only reporting changes without lowering changes

Tasks

1. Define critical activity family contracts
Create explicit pre-emission contracts for at least these families:
- Gmail send
- SMTP send
- Action Center / task creation and wait
- Retry wrappers / retry scopes
- Invoke workflow
- Data Service create/update/query
For each family, define:
- required properties
- valid property forms
- valid child-element/argument shapes
- disallowed pseudo-representations
- acceptable source spec patterns

2. Add family-aware lowering selection
Introduce a lowering step that maps structured workflow steps into one concrete family before emission.
Examples:
- Gmail send intent with Gmail connection/context should lower to Gmail family only
- SMTP send intent should lower to SMTP family only
- If the step is represented as narrative Try/Catch text plus a concrete send step, choose one canonical execution representation and lower the Try/Catch structure around it explicitly rather than treating literal prose as executable properties

3. Reject mixed-family drift before emission
If a workflow step or related step cluster ambiguously mixes:
- Gmail and SMTP send semantics
- pseudo Try/Catch literals and concrete activity definitions
- multiple incompatible activity packages for the same operation
then do not emit XAML and do not rely on post-assembly healing.
Instead, emit structured lowering diagnostics and fail package mode / allow DHG remediation.

4. Enforce critical required-property completeness before emission
For each chosen family, verify the contract before XAML emission.
Examples:
- Gmail send must have valid To/Subject/Body and family-specific connection context
- SMTP send must have valid To/Subject/Body and SMTP-family requirements
- Task creation must have valid required fields for the chosen task family
If the contract is incomplete:
- fail the workflow in package mode
- emit structured diagnostics
- allow DHG remediation
- do not emit placeholder-filled activity instances

5. Normalize pseudo-activity containers
Where the structured spec contains narrative container descriptions like:
- Try = "GmailSendMessage(...)"
- Catches = "Exception -> log ..."
these must not pass downstream as literal execution properties.
They must be either:
- lowered into a real TryCatch container with concrete child activities
- or rejected as non-emittable before XAML generation

6. Emit lowering diagnostics
Add a structured artifact:
`criticalActivityLoweringDiagnostics`
with per critical step:
- file/workflow
- source step/activity name
- detected intent
- selected activity family
- candidate families considered
- contractSatisfied: true/false
- missingRequiredProperties
- rejectedPseudoRepresentations
- packageFatal: true/false
- remediationHint

Also emit a compact summary:
- totalCriticalSteps
- totalLoweredSuccessfully
- totalRejectedForIncompleteContract
- totalMixedFamilyConflicts
- totalPseudoRepresentationRejections

7. Integrate with package-mode completeness policy
Wire lowering failures into the existing package completeness violation path so package mode fails early and honestly when critical family contracts are not satisfiable.

8. Add regression tests using current failure patterns
Cover:
- Gmail send step with Body present in spec lowers to a valid Gmail activity with Body preserved
- Gmail send intent does not produce SMTP fallback artifacts unless explicitly specified
- narrative Try/Catch send step is rejected or correctly lowered into real TryCatch + child activity
- critical send step missing Body is rejected before emission, not healed later
- Action Center create/wait steps with narrative pseudo-properties are rejected or lowered correctly
- Data Service update step with concrete entity object lowers correctly
- no placeholder injection occurs for rejected critical family contracts
- final package path no longer contains mixed Gmail/SMTP missing-Body artifacts for the same logical send intent
