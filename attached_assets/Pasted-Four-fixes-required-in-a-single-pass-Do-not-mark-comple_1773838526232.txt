FIX 2 — Double-wrapped InArgument
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The bracket wrapping logic is producing nested InArgument elements:

  WRONG:
  <InArgument x:TypeArguments="x:Object">
    <InArgument x:TypeArguments="x:Object">[Initialise]</InArgument>
  </InArgument>

  CORRECT:
  <InArgument x:TypeArguments="x:Object">[Initialise]</InArgument>

The wrapping logic is being applied twice — once during generation 
and once during the makeUiPathCompliant pass. Add a guard at every 
point where bracket wrapping runs:

  // Never wrap if content is already an InArgument or 
  // OutArgument element
  if (content.trim().startsWith('<InArgument') || 
      content.trim().startsWith('<OutArgument')) {
    return content  // already wrapped — do not touch
  }

This guard must be applied at ALL wrapping call sites — in the 
XAML generator, in makeUiPathCompliant, and in the catalog 
auto-correction layer. A wrapped element must never be wrapped 
again regardless of how many passes run over it.
