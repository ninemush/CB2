I want a dedicated upstream task for SDD/package-generation observability and timeout control.

Context
We had an SDD generation run take 300+ seconds, produce no valid artifacts block, and we could not reliably inspect what happened afterward because the relevant logs were not persisted. Current logging appears to rely primarily on stdout/console output, which is transient and not sufficient for debugging long-running generation flows.

This is a real operational gap:
- we need durable logs
- we need per-stage timing
- we need retry visibility
- we need fallback/degraded-state visibility
- we need enough timeout control to prevent long opaque hangs

Goal
Make generation runs observable and diagnosable after the fact, without relying on ephemeral stdout buffers.

What I want the task to cover

1. Persistent logging
- persist generation logs to durable server-side storage (file-based is acceptable if done cleanly)
- logs must survive stdout buffer rotation and restarts
- logs must be retrievable after the run completes or fails

2. Run correlation
- every generation run should have a clear run ID / correlation ID
- all log lines for that run should be traceable together

3. Per-stage timing
- log start, end, and duration for key stages such as:
  - platform capability fetch
  - prose generation
  - artifacts generation
  - artifact validation
  - artifact retries
  - fallback extraction
  - package generation
  - quality gate / validation
- timing should make long poles obvious

4. Retry/fallback visibility
- explicitly log:
  - retry count
  - why a retry occurred
  - whether fallback extraction ran
  - why it succeeded or failed
  - whether the final state was complete, degraded, blocked, or failed

5. Timeout control
- identify long-running LLM/platform calls that currently have weak or missing timeout policy
- introduce principled timeout controls where needed
- timeouts should fail truthfully and visibly, not silently hang

6. User-facing diagnosability
- the system should surface enough information to explain long-running or degraded outcomes
- not necessarily a full UI redesign, but at minimum the backend/run status should expose useful summary state

What I want back
Please return a task with:
1. title / objective
2. root-cause summary
3. impact analysis
4. done looks like
5. out of scope
6. tasks
7. relevant files
8. acceptance criteria

Important constraints
- keep this upstream
- do not solve this only with more console.log
- preserve existing generation behavior unless intentionally improved
- make the logging durable and useful for debugging real runs after the fact
- keep timeout behavior truthful and observable
