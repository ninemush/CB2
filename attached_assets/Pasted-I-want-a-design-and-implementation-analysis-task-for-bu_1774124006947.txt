I want a design and implementation-analysis task for building an official UiPath service endpoint/version matrix for cloud.uipath.com.

Important:
I do NOT want to use “inferred” as a user-facing state at all.

The rule should be:
- either we know the official endpoint and have successfully probed it
- or we have not
- and if we have not, we must know and surface the reason

So this task is about replacing vague endpoint assumptions with an explicit official endpoint/version matrix that MetadataService can own.

Goal
Create a canonical official UiPath service endpoint/version matrix for Automation Cloud (`cloud.uipath.com`) and use it as the basis for truthful probing, capability discovery, and UI status.

What I want from you

Phase 1 — Endpoint/Version Matrix Analysis
Please analyze the current system and produce a matrix that answers, for each UiPath service we care about:

1. Do we have an official documented endpoint for Automation Cloud?
2. What is the correct endpoint root/base path?
3. What is the service’s versioning model?
   - OData-style / no explicit vN in base path
   - explicit `/api/v1`
   - explicit query parameter like `api-version=1`
   - multiple supported official variants if applicable
4. Do we currently probe the correct official endpoint?
5. If not, what are we currently doing instead?
6. Can this service be marked as:
   - officially confirmed endpoint known
   - endpoint not officially confirmed yet
7. What probe should be used to confirm service availability on a tenant once the endpoint is known?

Please cover at minimum:
- Identity / token
- Orchestrator
- Test Manager
- Document Understanding
- Data Service / Data Fabric
- Integration Service
- Automation Ops
- Automation Hub
- Communications Mining
- IXP / Generative Extraction
- AI Center
- Maestro / PIMS
- Apps
- Assistant
- Agent Studio
- Autopilot
- Automation Store
- any other currently probed UiPath services in the codebase

Important:
I want this grounded in official UiPath documentation where possible.
If a service endpoint is not officially documented or cannot be confirmed from official docs, say that clearly.
Do not silently preserve “inferred” assumptions as if they were official.

Phase 2 — Status Model Redesign
Based on that matrix, define a new service-status model for the app.

I want to replace vague states like “inferred” with clear user-facing states.

Preferred state model:
1. Available
- official endpoint known
- official probe succeeded
- enough auth/scope exists to use/push

2. Unavailable (Endpoint Failure)
- official endpoint is known
- probe failed because endpoint/path/base is wrong, broken, deprecated, or unreachable in the expected form

3. Unavailable (Service Not Available)
- official endpoint is known
- probe confirms the tenant does not have the service enabled/provisioned/licensed

4. Unavailable (Authorization/Scope)
- official endpoint is known
- service exists/reachable but auth/scope/permission is insufficient

5. Unknown
- official endpoint not yet confirmed from docs
- or probe not yet run
- or evidence is incomplete

Please confirm whether this is the right model, or propose a better one — but it must avoid vague “inferred available” states.

Phase 3 — MetadataService Design
Then define how this should live in MetadataService without redundancy.

I want:
- MetadataService to own the official endpoint/version matrix
- no parallel source-of-truth
- no scattered hardcoded service roots outside MetadataService except internal emergency fallback logic if truly necessary

Please define:
1. what fields service-endpoints.json should contain for each service
2. how to represent:
   - official endpoint root
   - versioning model
   - official documentation status
   - probe path
   - expected success criteria
   - expected auth/scope model
3. how probe results should be recorded
4. how UI should consume those probe results truthfully

Phase 4 — Migration / Impact Analysis
Please analyze what it would take to adopt this model without breaking the app.

I want:
1. impacted files/modules
2. impacted flows:
   - connection test
   - diagnostics deep probe
   - capability discovery
   - TO-BE/PDD/SDD platform context
   - artifact generation decisions
   - deployment/provisioning
   - UI status display
3. migration sequence
4. regression risks
5. how to preserve app flow while making the service states more truthful

Deliverable

1. official endpoint/version matrix by service
2. explicit list of currently official vs currently unconfirmed services
3. proposed service status model
4. MetadataService schema/design changes needed
5. impact analysis
6. migration plan
7. acceptance criteria
