I want a focused design/analysis task for strengthening CannonBall’s XAML/dependency validation profile.

Important:
Do NOT implement yet.
I want analysis first.

Core principle
I do NOT want to replace the validations we have built from first-hand package generation failures.

I want the best of both:
- CannonBall’s lived, failure-driven validation and remediation logic
- UiPath’s stronger validation/analyzer structure and rule-model thinking

So this analysis should optimize for:
- preserving the hard-won validations we already have
- identifying where they should be formalized, reorganized, tightened, or moved earlier
- borrowing useful validation structure from UiPath concepts
- not overwriting our existing generation-aware checks unless there is a strong reason

Goal
I want to understand whether building a stronger, more explicit validation profile for generated UiPath packages would mainly extend/harden what we already have, or whether it would require replacing major parts of the current pipeline.

My belief is that this should mostly be an extension/hardening effort:
- we already have validation, quality-gate, dependency checks, and remediation logic
- but the rules are still incomplete, inconsistently layered, and sometimes too late in the pipeline
- many recent failures have clustered around XAML correctness and dependency authority

I want this validated against the current codebase.

What I want analyzed

1. Current validation architecture
Map the current validation / compliance / quality-check stack across package generation.

Please identify the existing layers, including things like:
- spec validation
- XAML structural validation
- semantic validation
- namespace/prefix validation
- dependency/package validation
- quality gate
- meta-validation
- remediation / repair passes
- pre-package validation
- any post-assembly validation

For each layer, describe:
- what it currently checks
- where it runs
- whether it prevents invalid output or only reacts after bad output is already generated
- whether it is one of our useful hard-won checks that should be preserved

2. Extend vs overwrite assessment
I want a clear assessment of whether a stronger validation profile would:
- mostly extend/harden the current architecture
- partially replace weaker/ad hoc logic
- or require major rewrites

Please identify:
- what should definitely remain
- what should be formalized
- what should be tightened
- what current logic is too brittle / redundant / late-stage and may need to be superseded
- what we should be careful not to lose because it came from real failures we’ve already encountered

3. Error pattern analysis
Based on the recent classes of failures we’ve seen, assess whether the majority of package-generation failures are clustering around:
- XAML emission correctness
- namespace/prefix correctness
- argument wrapper correctness
- InvokeWorkflowFile correctness
- dependency authority / package restore
- or somewhere else

I want to know whether a stronger validation profile is likely to materially reduce the dominant failure modes.

4. Comparison to UiPath-style validation structure
I do not need generic doc summaries.
I want an analysis of how UiPath-style validation/analyzer concepts could improve the organization of our existing validations.

Specifically:
- what structural ideas from UiPath would be useful for us
- what parts of our current validation are already stronger/more generation-aware than a generic analyzer model
- how we should combine the two without discarding our current real-world safeguards

5. Proposed validation profile
If this is worth doing, define the best validation profile shape for CannonBall.

I expect something like layered validation, for example:
- Layer 1: spec validation
- Layer 2: XAML structural validation
- Layer 3: semantic XAML validation
- Layer 4: dependency authority validation
- Layer 5: Studio-profile compatibility validation

Please recommend the real layer model based on the codebase, not just theory.

For each layer, specify:
- what should be validated
- whether it should be fail-fast, auto-repair, warning-only, or backstop
- what should happen upstream vs late-stage
- which existing CannonBall checks belong there

6. Preventative measures
Recommend the highest-value preventative checks / normalizers to add.

Examples:
- canonical activity-name/prefix normalization before XAML emission
- strict namespace coverage checks
- Assign argument-wrapper invariants
- InvokeWorkflowFile argument contract validation
- package authority enforcement
- Studio-openability compatibility checks
- regression corpus / representative package validation

Please distinguish:
- new checks we should add
- existing checks we should move earlier
- existing checks we should keep but formalize

7. Risk / breakage analysis
I want to know what introducing a stronger validation profile could break or complicate.

Please analyze risks like:
- duplicate validation across layers
- over-validating and rejecting packages that are currently usable
- fighting with current remediation passes
- moving checks earlier and changing behavior unexpectedly
- increasing pipeline latency too much
- accidentally removing useful hard-won validations while “cleaning things up”
- turning the system into a giant repair layer instead of a prevention layer

Please tell me clearly:
- is this worth doing now?
- would it mostly extend what we already have, or overwrite significant parts?
- what should definitely be preserved from the current system?
- should this become:
  - one implementation task
  - multiple linked tasks
  - or a phased hardening roadmap
