I want a new direction for CB2 focused on one outcome:

Produce a Studio-stable package that preserves business logic, even when full implementation is not yet safe.

This is NOT a request for another narrow unblock task.
This is NOT a request to keep optimizing toward full_implementation as the default.
I want a task set that repositions the pipeline around a safe, Studio-openable, business-logic-preserving output.

What I mean by “preserves business logic”
I do NOT mean “keep every emitted activity no matter what.”

I mean:
- preserve workflow boundaries
- preserve sequence/order of intended business steps
- preserve branching intent
- preserve retry intent
- preserve config/asset/integration intent
- preserve success/failure path intent
- preserve the meaning of a block even if the exact generated activity implementation is not safe

If a block cannot be safely emitted with verified activities/types/structure, I want the pipeline to replace that implementation with a stable handoff block that explicitly describes:
- what the block was supposed to do
- what inputs/outputs matter
- what business rule it represents
- what the developer must implement manually in Studio

Goal
The package should:
- open in Studio
- preserve the real business/process structure
- keep safe/proven logic as real generated XAML
- degrade unsafe/unverified logic into explicit handoff-safe blocks
- produce a DHG that clearly distinguishes:
  - preserved/generated logic
  - degraded/handoff blocks
  - manual work remaining

Context
Where we are now:
- CB2 has strong enforcement layers
- #371 / #372 and later tasks improved truthfulness and emission control
- but we still get blocked or degraded on core policy contradictions and structural issues
- full_implementation has not proven reliable as the default success target
- CB2YY appears better mainly because of its safer degradation and handoff posture, not because of a clearly superior generator core

What I want from you
Please propose the task set that turns CB2 into a:
“Studio-stable, business-logic-preserving compiler with explicit implementation handoff”

Answer these directly:
1. Is this the right target model for where CB2 is today?
2. What exactly needs to change in the pipeline to preserve business logic at the right abstraction level?
3. What should remain as real generated XAML?
4. What should degrade to handoff blocks?
5. What should still hard-block?
6. What 1-3 tasks would implement this shift?

Important principles I want you to assess
I think the right model is:

- baseline_openable becomes the true primary success mode for now
- success means Studio-openable + business-logic-preserving + truthful handoff
- full_implementation is attempted only for proven-safe patterns
- unsafe sections should not be silently removed or naively inner-stubbed
- unsafe sections should be replaced at the right granularity:
  - step
  - block
  - workflow
depending on what preserves the business meaning honestly
- DHG should become the explicit contract describing what was preserved vs what requires manual Studio implementation

Please tell me whether that framing is correct, partially correct, or wrong.

Output format
1. Short verdict
2. Target compiler model
3. What to preserve as generated logic
4. What to degrade into handoff blocks
5. What should still hard-block
6. Proposed next task set
For each task use:
- title
- what & why
- done looks like
- out of scope
- tasks
- relevant files
7. Why this will move us toward Studio-stable packages that still preserve business logic

Constraints
- Do not propose replacing CB2 wholesale with CB2YY
- Preserve CB2’s stronger enforcement architecture where it helps truth and stability
- Do not fall back to broad default-allow across the catalog
- Do not propose broad undeclared-variable auto-declaration
- Do not optimize first for “highest readiness score”
- Optimize first for:
  - Studio-openability
  - truthful package/dependency/type integrity
  - preserved business structure
  - bounded, explicit developer handoff

Be honest
If the right answer is “CB2 needs to become more like a compiler that emits safe handoff blocks for unsafe regions,” say that plainly.
If the right answer is “preserving business logic requires degrading at block/workflow granularity, not activity granularity,” say that plainly.
If the right answer is “full_implementation should stop being the default target for now,” say that plainly.
