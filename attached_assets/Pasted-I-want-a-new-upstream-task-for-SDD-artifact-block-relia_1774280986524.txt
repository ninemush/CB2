I want a new upstream task for SDD artifact-block reliability and readiness gating.

Context
We confirmed the latest package failure traces back to the SDD being saved without a valid `orchestrator_artifacts` block.

Current generation flow:
- one LLM call generates SDD prose
- one parallel LLM call generates Section 9 + `orchestrator_artifacts`
- `ensureArtifactBlock(...)` tries several fallbacks
- if all fail, the SDD can still be saved without artifacts

Observed failure:
- prose completed
- artifacts call failed to produce a valid structured block
- extraction fallback also failed
- final SDD was saved with no artifacts block

This is an upstream reliability issue.
Package generation should not be forced to infer deployment structure from prose when the structured deployment spec is missing.

What I want
Create a task that fixes artifact-block generation reliability and adds a clear readiness gate.

The task should cover:

1. Artifact-block reliability
- strengthen the dedicated artifacts-generation path so it produces the canonical fenced `orchestrator_artifacts` block more reliably
- reduce the chance of prose-only or partially structured output from the artifacts call

2. Fallback extraction robustness
- fix the extraction fallback so it is not crippled by overly aggressive input truncation
- ensure it has enough input context and output budget to extract a complete artifact block for realistic SDDs

3. Canonical validation
- validate the artifact block structurally before accepting it
- if invalid, classify why:
  - missing fence
  - invalid JSON
  - missing required sections
  - truncated/incomplete content

4. Readiness gating
- an SDD without a valid `orchestrator_artifacts` block should not be treated as package-ready
- the system should surface a truthful blocked state such as:
  - `SDD complete, deployment artifacts missing`
  - or equivalent
- do not silently proceed as if deployment provisioning will work normally

5. Downstream contract
- package/provisioning stages should consume the artifact block only when validated
- if the block is missing, downstream stages should either:
  - halt with a clear reason
  - or degrade explicitly and truthfully, not implicitly

What I want back
Please return a task with:
1. title / objective
2. root-cause summary
3. impact analysis
4. done looks like
5. out of scope
6. tasks
7. relevant files
8. acceptance criteria

Important constraints
- keep this upstream
- do not patch one SDD manually
- do not weaken correctness by guessing deployment artifacts from incomplete prose
- make the blocked/degraded state explicit and user-visible
