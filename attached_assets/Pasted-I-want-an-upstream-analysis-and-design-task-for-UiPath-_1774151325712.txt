I want an upstream analysis-and-design task for UiPath integration taxonomy, service/capability truthfulness, and connection UI simplification.

Important constraints:
- Do NOT introduce regression
- Do NOT remove any existing capability
- Do NOT create redundant metadata/source-of-truth layers
- Lean into MetadataService wherever it makes sense
- Keep the UI truthful, consistent, and intuitive
- I want impact analysis first, before any implementation plan is approved

Context
The current UiPath integration UI is mixing different concepts:
- External App resources
- UiPath services/products
- capabilities/features
- environment signals/observations

Examples of this confusion:
- IXP, Communications Mining, and Generative Extraction are treated too much like peer services
- Storage Buckets, Attended Robots, and Studio Projects are presented like services even though they behave more like capabilities/observations
- some cards are using token-acquired / scope-acquired / endpoint-accessible / provisioned as if they all mean “available”
- user-facing states are not yet cleanly aligned to what we actually know

The External Applications resource list in UiPath is useful backend truth, but it should not be surfaced directly as the user-facing service taxonomy.

Additional requirement:
I want the design to be grounded in the actual UiPath External Applications resources and current first-party service surfaces. We need a canonical mapping from:
- external app resource
- to user-facing service/product
- to capability/feature
without redundancy or ambiguity.
