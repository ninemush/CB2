I want an upstream architecture and implementation plan to harden UiPath package generation around real UiPath Studio compatibility and current UiPath activity/package definitions.

Context:
The latest generated package opens in Studio, but it still has serious upstream quality issues:
- Main.xaml is a stub fallback
- some generated activities/properties are inaccurate
- project.json contains dependency versions that Studio cannot restore from the configured feeds
- the current generation logic appears to rely on stale or weak assumptions about Studio compatibility and activity/package validity

I believe two missing upstream inputs are causing a lot of this:
1. The generator does not know the real/current UiPath Studio compatibility target strongly enough
2. The generator does not know the real/current UiPath activity package universe strongly enough

I want a concrete upstream solution for this.

Goal:
Make UiPath package generation depend on two formal, refreshable inputs:
1. a UiPath Studio Compatibility Profile
2. a UiPath Activity/Package Source of Truth

This should reduce:
- stale dependency versions
- invalid activity/property combinations
- package/feed mismatch
- drift between generated XAML and what current Studio/activity packages actually support

What I need you to design:
1. A UiPath Studio Compatibility Profile
2. A UiPath Activity/Package Source of Truth that can be refreshed independently of the generator code
3. A deterministic dependency selection process
4. A pre-emission validation step that blocks invalid activity/property/version combinations before XAML assembly
5. A clear integration plan with the existing:
   - activity catalog
   - workflow tree assembler
   - package assembler
   - package/dependency selection logic

Required architecture direction:
1. UiPath Studio Compatibility Profile
This should define things like:
- target framework
- Studio line/version family
- supported project type constraints
- allowed package version ranges
- compatibility rules for Windows vs Windows-Legacy vs Cross-platform if applicable
- minimum required activity package lines for the target profile

2. UiPath Activity/Package Source of Truth
This should be a public git-hosted file or set of files that CannonBall can consume.
It should define:
- package names
- allowed versions / preferred versions
- compatibility by Studio line and target framework
- activity definitions
- allowed properties
- required properties
- property types
- enum values where relevant
- activity-category metadata if useful
- any package-level constraints or feed notes

3. Deterministic dependency selection
The system must stop guessing dependency versions.
Instead it should:
- select package versions compatible with the target Studio Compatibility Profile
- prefer versions available from official/tenant feeds
- avoid emitting dependencies that are known not to resolve cleanly for the target profile

4. Pre-emission validation
Before XAML is emitted:
- validate that each chosen activity is allowed for the target profile
- validate property names and types
- validate required properties
- validate enum values
- reject or remediate invalid combinations before assembly
- do not let invalid activity/property/version combinations reach the package assembler

5. Integration with existing architecture
Preserve the current strengths:
- activity catalog
- tree-based workflow assembly
- quality gate
- remediation system

But strengthen the catalog into a typed executable contract, not just a prompt aid.

Important constraints:
- This is an upstream generation-quality fix, not a cosmetic patch
- The goal is not just that Studio opens the package
- The goal is that the generated package is structurally, dependency-wise, and activity-wise valid for the target UiPath Studio environment
- This plan should improve package quality without destabilizing the package builder architecture we are already fixing

Please deliver:
1. concise root-cause summary
2. proposed architecture changes
3. the exact purpose and contents of the UiPath Studio Compatibility Profile
4. the exact purpose and contents of the UiPath Activity/Package Source of Truth
5. how this integrates with the existing generator pipeline
6. how it should interact with activity catalog filtering and remediation
7. recommended implementation order
8. acceptance criteria

Important:
- Be concrete, not vague
- Focus on what will materially improve generated package correctness
- Preserve the package builder’s current architecture where possible
- Call out any parts of the current generation flow that should stop relying on inferred/stale package assumptions
