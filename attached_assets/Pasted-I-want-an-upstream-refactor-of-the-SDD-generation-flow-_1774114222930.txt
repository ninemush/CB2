I want an upstream refactor of the SDD generation flow to remove redundant logic and reduce fragility. This is not a patch task.

Problem:
The current SDD path has real redundancy in both message preparation and artifact extraction. Some of this duplication is already causing bugs (for example, inconsistent role filtering in chat-history-to-LLM prep), and some of it is making the SDD/orchestrator-artifacts flow slower and more fragile than it should be.

What I want:
Create an upstream refactor plan and implementation task that removes duplicated logic and establishes canonical shared functions for the SDD flow.

Key redundancies to address

1. Chat-history preparation is duplicated
Current state:
- the chat path has its own message-preparation/filtering logic
- document generation has trimChatForDocGen
- document revision has a separate raw history mapping path

This should become one canonical helper used by all LLM-facing paths.

2. Artifact parsing/extraction logic is duplicated
Current state:
- parseArtifactBlock exists as a helper
- the revision path partially reimplements the same regex/parse/fallback logic inline
- there are multiple artifact recovery attempts with overlapping responsibilities

This should become one canonical artifact parse/validate path.

3. Artifact generation/extraction has too many overlapping paths
Current state:
- dedicated artifacts generation call
- extraction from prose fallback
- recovery extraction call
- revision-time extraction if the block is missing

This is too much layered recovery logic around the same artifact payload.

4. History/data loading is repeated across the SDD/document lifecycle
Not every repeated load is bad, but I want you to identify where the same chat/document context is being fetched and reshaped redundantly within the same overall generation/revision workflow.

What I want back
Please deliver:

1. concise redundancy analysis
Group findings into:
- duplicated message preparation
- duplicated artifact parsing/extraction
- duplicated recovery logic
- repeated data loading/context shaping

2. upstream refactor task

- objective
- done looks like
- out of scope
- tasks
- relevant files

3. required refactor direction
The plan should include:
- one canonical prepareChatMessagesForLLM helper for all chat-history-to-LLM paths
- one canonical artifact parse/validate helper used by both generation and revision paths
- one primary structured artifact-generation path
- reduced fallback/recovery layering where possible
- clearer separation between:
  - SDD prose generation
  - artifact generation
  - artifact parsing/validation
  - revision-time artifact recovery

4. acceptance criteria
Must include:
- no duplicated ad hoc chat-history role filtering across chat/document/revision paths
- no duplicated inline artifact parsing logic when a shared helper exists
