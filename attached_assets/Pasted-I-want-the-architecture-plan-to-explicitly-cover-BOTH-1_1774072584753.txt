I want the architecture plan to explicitly cover BOTH:
1. out-of-band metadata refresh
2. runtime adoption of that metadata across the codebase

Context:
We already know CannonBall needs automatic freshness for:
- Studio compatibility metadata
- activity/package metadata
- UiPath integration endpoint metadata

But the refresh job alone is not enough.
Once the metadata exists, the system must be wired to actually use it in all the right places, replacing scattered hardcoded assumptions and stale inferred logic.

So I want a complete plan that covers:
- refresh
- snapshot storage
- runtime consumption/adoption

What I want:
Design a unified metadata architecture for CannonBall that covers:

A. Generation metadata
- UiPath Studio Compatibility Profile
- activity/package source-of-truth
- package version compatibility
- activity/property/enum compatibility

B. Integration metadata
- UiPath service endpoint map
- official vs inferred endpoint status
- API confidence levels
- service capability definitions
- deprecation / migration notes

And then define how the runtime must consume this metadata consistently.

Requirements:
1. Metadata must be refreshed automatically out of band on a scheduled basis
2. Runtime package generation must not perform live metadata discovery
3. Runtime integration logic must use local validated endpoint metadata, not scattered guessed strings
4. The system must use local last-known-good snapshots at runtime
5. If refresh fails, runtime continues on the last known-good snapshot
6. Freshness/staleness must be surfaced in logs and/or admin UI
7. Hardcoded stale fallbacks and scattered endpoint/version assumptions should be progressively replaced by metadata-backed lookups

What I want you to define:
1. overall architecture
2. metadata families
3. local snapshot file layout
4. what each file contains
5. official UiPath sources for refresh inputs
6. runtime consumption model for generation metadata
7. runtime consumption model for integration metadata
8. exactly which code paths should be rewired to use the metadata
9. failure/staleness behavior
10. phased implementation plan
11. acceptance criteria

- Studio Compatibility Profile selection

For integration metadata, explain how it should be wired into:
