Implement this as a CannonBall platform-level standard and architectural refactor, not as a one-off fix for a single automation or use case.

Mission:
CannonBall must generate first-time-pass, Studio-openable UiPath projects by default for all future applicable automations. This must be enforced centrally in the product, not left to prompt quality, individual task wording, or downstream cleanup.

Primary rule:
A minimal valid UiPath package is always better than a richer invalid package.

Core product objective:
For simple linear and API/data-driven automations, CannonBall must default to a minimal, internally consistent, Studio-openable UiPath project on the first pass before optimizing for richness, screenshots, diagnostics, REFramework patterns, or deployment sophistication.

What CannonBall is currently doing wrong:
- UiPath generation logic is duplicated across multiple routes and services instead of flowing through one canonical pipeline
- XAML generation, dependency aggregation, analyzer/fix passes, and Developer Handoff Guide generation are being invoked in more than one place
- Invalid output is being generated first and then “repaired” later in multiple layers
- The same bad patterns are being handled repeatedly instead of prevented once at source, including:
  - Workflows\ invoke path rewrites
  - ui:Assign tag rewrites
  - TakeScreenshot structural rewrites
  - placeholder stripping
  - dependency cleanup after over-generation
- The activity policy is fragmented:
  - the AI enrichment prompt permits risky activities
  - the activity registry marks them as valid
  - later packaging logic tries to repair or remove them
- This creates drift, regressions, and inconsistent behavior between generation paths

Platform-level behavior CannonBall should enforce:
1. For simple linear and API/data-driven automations, default to a flat minimal scaffold
2. Do not use REFramework unless the process explicitly requires queue/transaction execution
3. Do not emit hidden scaffold artifacts for simple/API automations
4. Do not emit any activity or property unless it is supported by the canonical validator schema
5. Build dependencies only from final surviving activities
6. Block packaging unless all hard quality-gate checks pass
7. If full implementation is risky, generate a clean valid baseline/stub instead of broken logic
8. Prefer a minimal valid package over a richer invalid one

Hard rules for simple/API automations:
- no Workflows\ path assumptions
- no REFramework workflows
- no ui:TakeScreenshot
- no ui:AddLogFields
- no placeholder tokens such as PLACEHOLDER or TODO markers
- no hardcoded credentials
- no dependency unless a final emitted activity requires it
- no activity that requires later post-processing to become valid

Add a “Studio-openable first” generation mode:
- Make this the default for simple-linear and api-data-driven automations
- This mode must prioritize:
  - valid XML/XAML
  - valid project.json
  - internally consistent workflow references
  - minimal required dependencies
  - valid stubs over broken business logic

Required architectural refactor:
1. Centralize all UiPath generation into one canonical pipeline
- all routes and chat flows must call the same orchestration layer
- no separate generation logic in multiple route handlers
- no duplicated “generate XAML + analyze + derive deps + build DHG” flows

2. Centralize scaffold selection
- create one pattern-based scaffold selector
- for simple-linear and api-data-driven automations:
  - use a flat scaffold
  - put XAML files at package root unless a folder structure is explicitly required
  - do not inject REFramework or hidden scaffold workflows
- only select REFramework for true transactional/queue processes

3. Centralize activity policy
- prompt/enrichment, generator, registry, validator, and packaging must all use the same canonical activity policy
- if an activity is not valid for a given automation pattern, it must not be emitted anywhere
- if an activity is not guaranteed to pass validation, it must not be emitted

4. Centralize output normalization
- normalization must happen once, in one place, before validation
- remove duplicate “repair” logic in separate modules
- do not rely on multiple downstream patch passes to make output valid

5. Make generation source-correct instead of repair-driven
- the generator must emit valid root-relative workflow paths directly
- the generator must emit valid activity tags directly
- the generator must emit valid dependencies directly
- the generator must emit valid stubs directly when confidence is low
- the pipeline must not depend on later rewrites of:
  - Workflows\ paths
  - ui:Assign tags
  - TakeScreenshot structure
  - placeholder removal
  - dependency pruning

6. Create explicit generation modes
- baseline_openable
- full_implementation
- if confidence is low or automation is simple/API-driven, prefer baseline_openable automatically

7. Add early safe-stub fallback
- if a workflow cannot be safely generated, emit a clean valid stub immediately
- do not wait for packaging or quality gate failure before falling back
- a stub must still be Studio-openable and correctly wired into the project

8. Build dependencies from final surviving activities only
- do not start from a broad dependency template
- do not inject packages “just in case”
- generate XAML first
- scan final XAML for actual activities
- derive project.json dependencies from the final surviving activity set only
- no dependency may be included unless at least one final activity requires it

Prompt/enrichment recommendations to implement centrally:
- update the enrichment prompt so simple/API automations do not emit:
  - ui:TakeScreenshot
  - ui:AddLogFields
  - UI automation activities unless UI work is explicitly required
  - mail activities unless the design truly uses them
  - placeholder-filled partial implementations where a valid stub would suffice
- if a workflow cannot be safely implemented, emit a valid stub workflow instead of broken logic
- the prompt should explicitly optimize for “Studio-openable correctness first”

Activity registry recommendations to implement centrally:
- maintain one canonical activity registry shared by generator and validator
- remove or gate risky activities for simple/API paths, especially:
  - ui:TakeScreenshot
  - ui:AddLogFields
  - any activity whose emitted schema is not consistently valid
- if an activity currently needs later rewriting to become valid, it should not be emitted directly

Hard quality gate before packaging:
Packaging must not begin unless all of the following pass:
- every XAML is valid XML
- every workflow reference resolves exactly as packaged
- every activity exists in declared dependencies
- every variable and argument is declared before use
- no placeholder or scaffold tokens remain
- project.json is valid
- dependency versions are compatible with target Studio profile
- archive manifest exactly matches expected output
- no hardcoded credentials or secrets
- no unknown activities
- no invalid activity properties

Recommended acceptance gate for “Studio-openable”:
1. every XAML parses as valid XML
2. every referenced workflow path exists exactly as packaged
3. every emitted activity is known to the declared dependency set
4. every variable and argument is declared before use
5. no placeholder or scaffold tokens remain
6. dependency versions are compatible with the selected UiPath Studio target
7. packaged file list exactly matches the raw manifest
8. no hardcoded credential values
9. no unknown activity identifiers
10. no invalid activity properties

Regression coverage required:
Add tests/fixtures that fail if CannonBall reintroduces:
- Workflows\ invoke mismatch
- invalid XML/XAML
- ui:TakeScreenshot in simple/API automations
- ui:AddLogFields in simple/API automations
- undeclared variables or arguments
- placeholder leakage
- dependency over-injection
- malformed project.json
- archive manifest / XAML parity mismatch
- generator/validator schema drift

Specific implementation expectation:
Review and refactor the current code paths governing:
- AI enrichment prompt and activity normalization
- activity registry
- XAML generation
- dependency assembly
- package assembly
- quality gate validation
- fallback generation logic
- all route-level generation entry points
- all chat-triggered generation entry points

Expected end state:
CannonBall should no longer have multiple places that generate UiPath artifacts differently or patch the same problems repeatedly. It should have one authoritative generation pipeline that:
- selects the right scaffold by automation pattern
- emits only valid supported activities
- derives only necessary dependencies
- validates before packaging
- falls back early to clean stubs when needed
- produces first-time-pass, Studio-openable UiPath output by default

Important:
- Do not treat recent UiPath failures as use-case-specific bugs
- Do not solve this with prompt wording alone
- Do not optimize for being impressive; optimize for being correct, openable, and internally consistent first
- Implement this as a CannonBall-wide generation standard enforced in code
