Integration Service First-Class Connector Lowering and Mail Source-of-Truth Unification

What & Why
The latest runs prove that the mail-send problem is not primarily missing property knowledge. The structured spec already contains:
- a narrative TryCatch describing summary email send via the Gmail Integration Service connector using InGmailConnectionName
- a separate concrete GmailSendMessage step from UiPath.GSuite.Activities with To, Subject, and Body=summaryBody

Those are two competing representations of the same logical send operation. Later stages still drift further into generic Mail / SMTP-family artifacts, producing:
- Send SMTP Mail Message missing required Body
- Send Gmail Message missing required Body
- UiPath.Mail.Activities appearing in the package surface for a Gmail Integration Service use case

This task makes Integration Service mail intent a first-class canonical source of truth when the spec/use case indicates a connector-based send. It unifies mail source selection before emission, eliminates competing mail representations in the same logical cluster, and prevents downstream drift into GSuite or generic Mail/SMTP artifacts unless explicitly chosen.

Goals
1. When the use case/spec indicates Integration Service mail sending, make Integration Service the canonical source of truth for that logical send cluster.
2. Prevent one logical send cluster from simultaneously carrying:
   - Integration Service narrative intent
   - GSuite concrete activity intent
   - generic Mail / SMTP fallback artifacts
3. Lower Integration Service mail intent into exactly one valid executable representation for the active Studio target/profile, or fail before emission.
4. Preserve required runtime properties like To, Subject, and Body through lowering.
5. Fail early and honestly if the cluster cannot be mapped to one valid concrete execution model.

Done looks like
- A logical mail-send cluster with Integration Service intent is identified before XAML emission.
- If Integration Service intent is present and selected as canonical, the cluster may not also emit GSuite or SMTP-family activities for that same logical operation.
- If the pipeline has no valid concrete executable mapping for Integration Service mail under the active target/profile, the cluster is rejected before emission with structured diagnostics.
- Narrative TryCatch send descriptions are either:
  - lowered into real TryCatch containers around the canonical Integration Service execution representation
  - or rejected before emission
- Required properties (To, Subject, Body, connection context) are preserved through lowering.
- No wrong-family package/activity tags appear later for an Integration Service-locked cluster.
- A structured diagnostics artifact records canonical source selection and any rejected competing representations.

Out of scope
- Broad catalog/filter alignment
- General parity/freeze changes
- Full decomposition redesign
- Non-mail family source-of-truth work
- UI-only reporting changes

Tasks

1. Add canonical mail source selection
Introduce a mail source-of-truth selection step for each logical mail-send cluster.
Candidate sources may include:
- Integration Service connector intent
- GSuite activity intent
- generic Mail/SMTP intent
If the spec/use case includes Integration Service connection intent for the cluster, that source must be preferred and locked unless an explicit override is present.

2. Detect and reject competing mail representations
Within the same logical cluster, detect simultaneous presence of:
- Integration Service narrative send intent
- concrete GmailSendMessage / GSuite activity nodes
- generic Mail/SMTP-family nodes
- narrative TryCatch text describing the same send
If multiple sources survive for the same logical operation, do not emit them all. Either:
- convert them into one canonical execution model
- or reject the cluster before emission

3. Add Integration Service concrete lowering
Implement one canonical lowering path for connector-based mail-send intent.
The lowering step must resolve to the exact concrete executable representation valid for the active Studio target/profile and verified dependency set.
If no valid concrete representation exists under the active target/profile, fail before emission with structured diagnostics rather than falling back silently to GSuite or SMTP.

4. Preserve required properties through canonical lowering
If the source cluster contains To, Subject, Body, and connection name/context, ensure the chosen canonical representation preserves them.
If Body exists in the source spec but is lost during lowering, treat that as a lowering failure and fail before emission.
Do not allow placeholder injection for rejected connector-locked clusters.

5. Eliminate narrative container leakage
Narrative TryCatch properties such as:
- Try = "GmailSendMessage(...) via Integration Service connection ..."
- Catches = "Exception -> ..."
must not survive as executable payloads.
They must be either:
- lowered into a real TryCatch container around the canonical connector execution step
- or rejected before emission

6. Add downstream source-lock guardrails
Once a cluster is locked to Integration Service mail intent:
- no GSuite activity tags may appear later for that cluster unless explicitly selected
- no UiPath.Mail.Activities / SMTP-family artifacts may appear later for that cluster
- dependency analysis and package assembly must treat wrong-family appearance as package-fatal source-drift, not a late-discovery warning

7. Emit diagnostics
Add:
`mailSourceOfTruthDiagnostics`
with per cluster:
- workflow/file
- clusterId
- detectedMailSources
- selectedCanonicalSource
- selectedConcreteType
- selectedConcretePackage
- target/profile compatibility
- requiredPropertiesSeen
- missingRequiredProperties
- rejectedCompetingRepresentations
- packageFatal
- remediationHint

Also include summary counts:
- totalMailClusters
- totalIntegrationServiceLocked
- totalRejectedForCompetingSources
- totalRejectedForNoConcreteConnectorMapping
- totalBodyLossDuringLowering
- totalWrongFamilyDriftBlocked

8. Integrate with package completeness policy
Rejected or incomplete connector-locked mail clusters must flow into packageCompletenessViolations and fail package mode early and honestly.

9. Regression tests
Cover:
- Integration Service mail cluster with To/Subject/Body lowers to one canonical connector-based execution representation
- Integration Service cluster does not also emit GSuite activity
- Integration Service cluster does not later introduce UiPath.Mail.Activities / SMTP artifacts
- narrative TryCatch + connector send is correctly lowered or rejected
- Body present in source spec is preserved; if lost, lowering fails before emission
- no placeholder injection for rejected connector-locked clusters
