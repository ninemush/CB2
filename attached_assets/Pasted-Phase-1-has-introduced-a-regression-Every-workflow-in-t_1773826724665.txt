The rule: any attribute or child element that belongs to a 
namespace OTHER than the activity's own package namespace 
(ui:, System.Activities, etc.) should be treated as framework 
metadata and skipped by the validator entirely. The validator 
should only check properties that belong to the activity's 
own namespace.

FIX 2 — CATALOG_VIOLATION must never trigger STUB_BLOCKING_FALLBACK

This is the most important fix. A CATALOG_VIOLATION is a quality 
signal, not a reason to destroy the workflow.

The stub fallback logic must be changed so that:
- CATALOG_VIOLATION errors are treated as WARNINGS, not blocking 
  errors, until the catalog has been proven reliable in production
- STUB_BLOCKING_FALLBACK is only triggered by genuinely structural 
  failures: malformed XML, zero activities in a workflow, or a 
  complete generation failure
- The quality gate severity for CATALOG_VIOLATION must be set to 
  "warn" not "error" for now

This is a deliberate product decision: the catalog validator is 
new and its false positive rate is unknown. Making it fatal 
immediately was wrong. It should surface violations as warnings 
visible in the UI (READY_WITH_WARNINGS state from Task B) and 
allow the developer to review them — not silently destroy every 
workflow.
