#112 - Pipeline Survivability Architecture
What & Why
Redesign the UiPath package generation pipeline around a progressive convergence model so it almost never ends in "no package generated." Today, three fail-closed paths still exist:

In full_implementation mode, if the quality gate fails after auto-remediation, buildNuGetPackage throws a QualityGateError (line 1693 of uipath-integration.ts) that kills the entire build with no fallback attempt.
If enrichWithAITree in uipath-integration.ts fails, times out, or returns a malformed WorkflowSpec, the pipeline can produce no usable XAML and end with an empty or missing package.
If buildNuGetPackage produces an empty buffer (e.g., ZIP archive creation fails), the pipeline declares FAILED with no recovery attempt.
This task eliminates all three paths by making every failure converge through the same canonical generateUiPathPackage entrypoint using automatic downgrade to baseline_openable, and by adding a new FALLBACK_READY status so the system can clearly distinguish degraded-success from clean-success or total failure.

All downgrade/retry paths must go through the existing canonical generateUiPathPackage function in uipath-pipeline.ts — no side entrypoints or parallel build paths.

Done looks like
When full_implementation fails quality gate remediation, the pipeline catches the QualityGateError, records a typed DowngradeEvent in pipeline warnings, and retries through the same generateUiPathPackage entrypoint with baseline_openable mode — no unhandled throw kills the build
A maxDowngradeAttempts guard (default: 1) prevents recursive downgrade loops — if baseline_openable also fails, the pipeline declares FAILED cleanly
When enrichWithAITree fails, times out, or returns unparseable output, the pipeline falls back to deterministic scaffold generation using approved process map nodes + approved SDD/PDD context when present, or map-only scaffold when document context is absent or weak
FALLBACK_READY is a first-class value in the PackageStatus type alongside READY, READY_WITH_WARNINGS, and FAILED — it is set whenever the pipeline converged through auto-downgrade or AI enrichment fallback
All downgrade events are captured as typed DowngradeEvent objects (not generic PipelineWarning strings) with fields: fromMode, toMode, reason, triggerStage, timestamp
The pipeline cache key includes whether the result was produced via downgrade, so a FALLBACK_READY result is not served from cache as if it were a READY result for the same ideaId
The "structurally invalid or non-emittable archive" failure condition is broadened: if buildNuGetPackage produces an empty buffer, a corrupted ZIP, or an archive missing Main.xaml or project.json, the pipeline retries once with baseline_openable before declaring FAILED
A pre-assembly catalog property filter strips activity properties not present in the activity catalog from the WorkflowSpec tree before it reaches assembleWorkflowFromSpec, so non-catalog properties never produce quality gate violations
The front-end UiPathPackageCard in document-card.tsx handles the FALLBACK_READY status with distinct UI language: amber indicator, message like "Package generated with reduced scope — see Developer Handoff Guide for details", clearly distinguishable from READY_WITH_WARNINGS
Total package failure (FAILED) is reserved for truly catastrophic conditions: ZIP archive cannot be created, baseline_openable retry also fails, or the process map has zero nodes

Out of scope
Per-activity or per-sequence granular fallback within workflows (separate task)
Enhanced DHG reporting of auto-fix/stub/downgrade actions (separate task)
Changes to the UiPath deployment or download endpoints
Changes to quality gate rules or meta-validator correction categories
UI redesign beyond handling the new FALLBACK_READY status

#113 - Granular Fallback and DHG Reporting
What & Why
Today, when a workflow has invalid activities, the entire workflow is replaced with a stub via generateStubWorkflow. This discards all valid logic — variables, arguments, control flow, and surrounding sequences — in the workflow. The remediation path in uipath-integration.ts (lines 1604-1672) operates at whole-workflow granularity only.

Additionally, the auto-fix actions (autoFixSummary), stub insertions (earlyStubFallbacks), quality issues (collectedQualityIssues), and downgrade decisions made during the pipeline are logged to console but not systematically fed into the Developer Handoff Guide. The DHG currently accepts only stubbedWorkflows?: string[] and qualityIssues?: DhgQualityIssue[] — it has no structured input for remediation actions, their granularity, or the original intent of replaced components.

This task adds per-activity and per-sequence fallback granularity and introduces a canonical PipelineOutcomeReport object that flows from the build engine into DHG generation as the single source of truth.

Done looks like
When a specific activity within a workflow fails validation (as identified by ClassifiedIssue from classifyQualityIssues — not ad hoc string matching), only that activity is replaced with a ui:Comment stub that preserves the original activity type, DisplayName, and the blocking reason — the surrounding sequence structure, workflow arguments, variables, and control-flow shape (TryCatch, If, ForEach) remain intact
When multiple activities within a sequence are all invalid but the sequence boundaries and parent control flow are valid, the sequence content is replaced with a stub sequence that documents the intended behavior — the parent workflow structure and arguments are preserved
Whole-workflow stub replacement via generateStubWorkflow only happens as a last resort when per-activity and per-sequence repair still cannot produce valid XAML (verified by re-running runQualityGate after each repair level)
The escalation ladder is explicit: per-activity → per-sequence → per-workflow, with re-validation at each level before escalating
A new PipelineOutcomeReport interface captures all remediation actions as typed, structured entries: { level: "activity" | "sequence" | "workflow", file: string, originalTag?: string, originalDisplayName?: string, reason: string, classifiedCheck: string, developerAction: string, estimatedEffortMinutes: number }
PipelineOutcomeReport also includes sections for auto-repairs (from autoFixSummary), downgrade events (from DowngradeEvent[] introduced in the survivability task), and quality gate warnings — replacing the current practice of passing loose arrays
DhgOptions in xaml-generator.ts is extended to accept a PipelineOutcomeReport instead of the current separate stubbedWorkflows, qualityIssues, and generationModeReason fields (the old fields remain supported for backward compatibility)
The DHG output includes new sections organized by remediation category:
Fully Generated — workflows that passed quality gate without intervention
Auto-Repaired — specific fixes applied (what was fixed, in which file, how)
Stubbed — items replaced at activity, sequence, or workflow level, with original intent, blocking reason, and developer action needed
Downgraded — mode changes from full to baseline, with trigger reason (from DowngradeEvent)
Each stubbed item in the DHG includes a concrete developer action item with estimated effort (e.g., "Recapture UI selector for Click activity in ProcessInvoice.xaml — estimated 15 min")
The front-end UiPath package card shows a remediation summary count when the package is READY_WITH_WARNINGS or FALLBACK_READY (e.g., "3 activities stubbed, 2 auto-repaired — see DHG")
Invalid activity detection in the remediation path uses ClassifiedIssue.check and ClassifiedIssue.file from classifyQualityIssues to locate the failing activity tag within the XAML, rather than ad hoc regex or string matching against violation detail text

Out of scope
Changes to the pipeline convergence model, PackageStatus types, or downgrade retry logic (covered by the survivability task)
Changes to the quality gate rules themselves or adding new violation checks
Changes to the meta-validator correction categories
Changes to deployment or download endpoints

Implement per-activity stub replacement: in the quality gate remediation path in uipath-integration.ts, when classifyQualityIssues identifies a blocking issue tied to a specific activity (via ClassifiedIssue.file and ClassifiedIssue.check), locate that activity tag in the XAML and replace it with a ui:Comment stub documenting the original activity type, DisplayName, and reason.
