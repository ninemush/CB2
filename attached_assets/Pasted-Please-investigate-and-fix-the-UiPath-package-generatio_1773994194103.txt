Please investigate and fix the UiPath package generation trigger race condition in CannonBall.

Observed behavior:
- User sends: "generate uipath package"
- Server correctly classifies the intent as UIPATH_GEN
- Server correctly sends the acknowledgement token:
  "Starting UiPath package generation..."
- Server correctly sends triggerUiPathGen: true
- Chat stream completes normally
- But the client never actually calls POST /api/ideas/:ideaId/generate-uipath
- UI gets stuck showing a generic thinking message like "Getting things ready..."
- Server logs confirm the generation API is never invoked

Current suspected cause:
- In the workspace chat client, pendingUiPathGenRef.current is set when triggerUiPathGen is received
- After the stream ends, the client tries to call generateUiPathRef.current?.(true) after a short timeout
- But generateUiPath() has a guard like:
  if (isGeneratingDoc || isStreaming) return;
- At the moment the delayed call fires, one of those states is still true, so the function silently bails out
- Result: the trigger is lost and package generation never starts

What I want:
Implement a durable, race-safe trigger mechanism for UiPath generation so the trigger cannot be lost just because a state flag has not cleared yet.

Requirements:
1. Do not rely on a fragile one-shot timeout to trigger generateUiPath().
2. If triggerUiPathGen is received while isStreaming or isGeneratingDoc is still true, the trigger must remain pending and execute automatically once the blocking state clears.
3. The trigger must be idempotent:
   - do not start multiple generate-uipath requests
   - do not lose the trigger
4. The UI should show a correct package-generation-specific progress state, not a misleading generic stage-thinking state, once the generation trigger is pending or running.
5. Preserve the existing product behavior:
   - chat acknowledges first
   - then generation starts automatically
6. Add clear logging or instrumentation around:
   - trigger received
   - trigger deferred
   - trigger executed
   - generate-uipath request started
7. Do not implement a patchy timeout increase. Fix the state orchestration properly.

Implementation direction:
- Review the workspace chat flow where triggerUiPathGen is handled
- Replace the fragile timeout-based handoff with a durable pending-generation state/effect
- The effect should watch the relevant blocking conditions and fire generation once they are clear
- Make sure generateUiPath() distinguishes:
  - generation already running
  - trigger pending
  - trigger consumed
- Ensure the pending trigger is only cleared after POST /api/ideas/:ideaId/generate-uipath is actually started

Files to inspect first:
- client/src/pages/workspace.tsx
- especially the chat stream completion flow
- triggerUiPathGen handling
- generateUiPathRef / generateUiPath()
- any progress indicator state tied to isStreaming / isGeneratingDoc / package generation

Deliverable:
1. Brief diagnosis of the exact race condition
2. The fix
3. Short explanation of why the trigger can no longer be lost
4. Any follow-up UX improvements needed for progress text
