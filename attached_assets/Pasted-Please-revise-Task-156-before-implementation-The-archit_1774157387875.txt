Important correction to the current #156 proposal
OIDC discovery should be the source of truth for valid OAuth scope families/resources.
It should NOT be treated as the sole source of truth for:
- whether a public API exists
- whether a service should appear in the user-facing taxonomy
- whether a capability matters to TO-BE / PDD / SDD / deployment decisions

So the revised architecture must explicitly separate:

1. Scope/Auth model
- dynamically derived from OIDC discovery (`scopes_supported`)
- answers:
  - which scope families/resources are real
  - which scopes should be requested for tokens

2. Endpoint/API model
- docs-backed / metadata-backed service endpoint truth
- answers:
  - official endpoint root/base path
  - versioning model
  - whether a public API is officially supported
  - allowed alternates/fallbacks
  - deprecation state
  - last verified timestamp/source

3. Product/Capability taxonomy model
- service/product vs capability/feature vs observation/signal
- answers:
  - what the user sees
  - parent/child relationships
  - supported external API vs unsupported external API
  - whether a thing should still appear in the UI/context even if it lacks standalone OAuth scopes

- service-root URLs

D. Graceful degradation
Revise the task so metadata or probe failures do NOT take down the whole app.

Required degradation behavior:
- if OIDC discovery fails, use last-known-good snapshot and mark stale
- if one service endpoint fails, degrade only that service/capability
- if a service probe fails, do not fail TO-BE / PDD / SDD / package generation / deployment globally
- OR/package/deployment flows must continue if Orchestrator remains healthy

E. Endpoint revision / deprecation handling
Add explicit support for API revision and deprecation handling.

The revised task must define how MetadataService will model:
- current endpoint
- alternate/fallback endpoints where appropriate
- versioning model
- deprecation state
- last verified timestamp
- verification source

And how the runtime should behave when:
- a previously working endpoint starts failing
- a deprecated endpoint is detected
- an alternate endpoint succeeds
- no valid endpoint works

The desired behavior is:
- switch to allowed alternate when safe
- mark the service as degraded with exact reason
- emit telemetry/warnings
- avoid whole-system failure

F. Preserve capability visibility
Do NOT remove capabilities from the user-facing taxonomy just because they lack standalone OAuth scope families or public APIs.

Instead classify them truthfully as one of:
- supported external API
- unsupported external API
- capability under parent service
- observation/signal

Examples:
- Apps may remain visible but classify as Unsupported External API if that is the truth
- Communications Mining / Generative Extraction may remain visible as capabilities under the appropriate parent
- observations like Attended Robots / Studio Projects should not disappear if they still matter

- what scopes are valid
- what scopes were granted
- what is missing for a given service/capability
- what the user should do next
