#112 - Pipeline Survivability Architecture
What & Why
Redesign the UiPath package generation pipeline around a progressive convergence model so it almost never ends in "no package generated." Today, the pipeline can still throw a QualityGateError in full_implementation mode that kills the entire build, AI enrichment failures can produce no output, and there is no explicit degraded-success status. This task eliminates those fail-closed paths and enforces deterministic boundaries between LLM enrichment and catalog-backed assembly.

Done looks like
When full_implementation fails quality gate remediation, the pipeline automatically retries with baseline_openable and reports the downgrade — no more thrown QualityGateError killing the build
When AI enrichment (ai-xaml-enricher.ts) fails or returns malformed output, the pipeline falls back to deterministic scaffold generation from the process map nodes + SDD context instead of producing no package
A new FALLBACK_READY status exists alongside READY and READY_WITH_WARNINGS, representing a structurally valid but degraded package
The front-end handles the FALLBACK_READY state with appropriate messaging (e.g., "Package generated with fallback — see DHG for details")
Activity properties emitted by the LLM that are not in the activity catalog are stripped or normalized before reaching the quality gate, preventing false violations
The empty-buffer / no-package path is eliminated — if assembly fails, the pipeline retries with baseline_openable mode before giving up
Total package failure (FAILED) is reserved only for truly catastrophic structural corruption (e.g., ZIP archive cannot be created at all)

Out of scope
Per-activity or per-sequence granular fallback (separate task)
Enhanced DHG reporting of auto-fix/stub actions (separate task)
Changes to the UiPath deployment or download paths (those remain unchanged)
UI redesign beyond handling the new FALLBACK_READY status

Tasks
Wrap the QualityGateError throw path in uipath-integration.ts with auto-downgrade logic: catch the error, retry with baseline_openable mode, and record the downgrade event in pipeline warnings.
Add AI enrichment fallback: wrap the enrichWithAITree call path so that if the LLM fails, times out, or returns unparseable output, the pipeline generates a deterministic workflow scaffold from the process map nodes and SDD context.
Add FALLBACK_READY as a first-class PackageStatus value in the pipeline types, and set it when the pipeline converged through auto-downgrade or AI enrichment fallback.
Add a pre-assembly catalog property filter: after the LLM produces the workflow tree spec, strip any activity properties not present in the activity catalog before passing to the tree assembler, so non-catalog properties never reach the quality gate.
Eliminate the empty-buffer failure path: if buildNuGetPackage produces an empty buffer, retry once with baseline_openable mode before declaring FAILED.
Update the front-end to handle the FALLBACK_READY status — show a distinct message indicating the package was generated with fallback, with a link to the DHG for details.

#113 - Granular Fallback and DHG Reporting
What & Why
Today, when a workflow has invalid activities, the entire workflow is replaced with a stub. This discards all valid logic in the workflow. Additionally, the auto-fix actions, stub insertions, and downgrade decisions made during the pipeline are logged to console but not surfaced in the Developer Handoff Guide (DHG). This task adds per-activity and per-sequence fallback granularity and makes the DHG the complete truth layer for everything that happened during generation.

Done looks like
When a specific activity within a workflow fails validation, only that activity is replaced with a Comment stub preserving a description of what was intended — the rest of the workflow's valid sequences, variables, and logic remain intact
When a full sequence (not just one activity) is invalid but the rest of the workflow is fine, only that sequence is replaced with a stub sequence
Whole-workflow stub replacement only happens as a last resort when per-activity and per-sequence repair cannot produce valid XAML
The DHG explicitly reports every action taken during generation, organized into clear categories:
Fully generated (Tier 1)
Auto-repaired with specific details of what was fixed
Downgraded from full to baseline with reason
Stubbed at activity, sequence, or workflow level with reason
What exact developer action is needed for each stubbed/repaired item
The autoFixSummary array and earlyStubFallbacks list from the build process are fed into the DHG generation function
Each stubbed activity Comment includes the original activity type, intended purpose, and why it was replaced

Out of scope
Changes to the pipeline convergence model or status types (covered by the survivability task)
Changes to the quality gate rules themselves
Changes to the meta-validator correction categories

Tasks
Implement per-activity stub replacement in the quality gate remediation path: when a specific activity fails validation, replace only that activity tag with a ui:Comment stub that documents the original intent, preserving the surrounding sequence structure.
Implement per-sequence stub replacement: when multiple activities within a sequence are invalid but the sequence boundaries are clear, replace the sequence content with a stub sequence that documents what was intended.
Update the escalation ladder in uipath-integration.ts so remediation tries per-activity first, then per-sequence, then per-workflow stub replacement.
Pass autoFixSummary, earlyStubFallbacks, collectedQualityIssues, and any downgrade events into the DHG generation function as structured input.
Add new DHG sections that report auto-repaired items (what was fixed and how), stubbed items at each granularity level (activity/sequence/workflow, with reason and original intent), and downgrade events (from full to baseline, with trigger reason).
For each stubbed item in the DHG, include a concrete developer action item with estimated effort.
