Required Property Binding Enforcement and Structured Expression Lowering Before Compliance

What & Why
After archive-authority unification, the pipeline is no longer primarily failing on freeze/parity/source-of-truth architecture. The latest package now surfaces a narrower but still systemic defect family:

- required activity properties remain unresolved and are being auto-filled with PLACEHOLDER-style pseudo-XAML
- structured expressions are still not consistently lowered into valid UiPath/XAML forms before compliance and final validation
- the pipeline still degrades around unresolved required properties instead of resolving them deterministically or failing earlier in package mode

This is not just a mail issue. Mail Body is the current visible symptom, but the broader product problem is:
when a required activity property or structured expression cannot be resolved, the pipeline falls back to placeholder injection or malformed content rather than enforcing valid binding or explicit package-mode failure/handoff.

This task introduces a systemic fix for that defect class.

Goals
1. Resolve required properties deterministically where possible before compliance/final validation.
2. Prevent placeholder injection into final package-mode XAML for unresolved required properties.
3. Lower structured expressions into valid UiPath/XAML forms before compliance, or preserve them as explicit structured defects without emitting pseudo-XAML.
4. Apply this generically across activity families, not just mail.

Done looks like
- Required properties across supported activity families are validated and bound before final compliance/validation.
- If a required property cannot be bound deterministically, package mode does not inject PLACEHOLDER into final XAML.
- Structured expressions for target/value/content-like properties are lowered into valid UiPath/XAML forms before compliance, or the workflow is marked with a structured defect without emitting malformed/pseudo-XAML.
- The latest visible failure pattern (required Body on send activity) is eliminated as a manifestation of the broader rule, not via a one-off patch.
- Final-quality reporting distinguishes:
  - resolved required-property bindings
  - unresolved required-property defects
  - structured-expression lowering failures
  - degradation substitutes, if any
- Build mode behavior is preserved except that invalid placeholder fallback in package mode is no longer allowed.

Out of scope
- Broad catalog/filter alignment
- New freeze/parity/source-of-truth architecture
- Business-logic invention
- Reworking decomposition strategy
- Special-casing only mail activities

Tasks

1. Create a generic required-property enforcement layer
Introduce a validation/binding layer that operates before compliance/final validation and checks required activity properties generically across supported activity families.
For each required property:
- determine whether a valid source binding exists
- bind it deterministically if available
- if not available, record a structured unresolved-required-property defect
- do not emit PLACEHOLDER into final package-mode XAML

2. Generalize beyond one activity family
Implement this as a property-contract rule, not a mail-specific patch.
The layer should work for any activity with required properties, including but not limited to:
- content/body fields
- title/subject/message fields
- path/file fields
- queue/task/form required properties
- other required target/value/content-like slots

3. Add structured expression lowering before compliance
Introduce or strengthen a pre-compliance lowering step for structured expressions and value objects so they become valid UiPath/XAML forms before compliance normalization.
If lowering is deterministic, emit the valid final form.
If lowering is not deterministic:
- record a structured defect
- do not emit pseudo-XAML or malformed placeholder content
- degrade or fail according to existing package-mode rules

4. Eliminate placeholder injection in package mode
Remove or block any package-mode path that auto-injects PLACEHOLDER into unresolved required executable properties.
If a required property is unresolved:
- either bind a valid deterministic value
- or mark as structured defect / package-fatal / handoff-required according to existing mode semantics
- but never emit pseudo-XAML placeholder values as if runnable

5. Add diagnostics
Emit structured diagnostics such as:
- requiredPropertyBindings[]
- unresolvedRequiredPropertyDefects[]
- expressionLoweringFixes[]
- expressionLoweringFailures[]

Each record should include:
- file
- workflow
- activityType
- propertyName
- sourceBinding or failureReason
- originalValue
- resolvedValue if applicable
- severity
- packageModeOutcome

6. Add package-mode guard
Before final compliance/validation, assert that no required executable property scheduled for final XAML emission still contains:
- PLACEHOLDER
- TODO
- STUB
- HANDOFF
- unresolved synthetic sentinels
If found, fail/degrade through the structured defect path rather than emitting pseudo-XAML.

7. Add regression tests
Cover:
- required Body-like property successfully bound from real upstream value
- unresolved required content/body property does not emit PLACEHOLDER
- required property in another non-mail activity family also respects the rule
- deterministic structured-expression lowering success
- non-deterministic structured-expression lowering becomes structured defect, not malformed XAML
- final package-mode XAML contains no required-property PLACEHOLDER emission
- normalized file mapping and reporting still work
