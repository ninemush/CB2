Required Property Loss Root-Cause Trace and Enforcement Path Investigation

What & Why
Recent benchmark runs show that some critical activities still fail because required runtime properties do not survive all the way into final XAML. The clearest current examples are:
- GmailSendMessage.Body
- UpdateEntity.EntityObject

The codebase already appears to contain substantial required-property infrastructure:
- required-property-enforcer.ts for precedence resolution, provenance tracking, sentinel detection, and binding logic
- emission-gate.ts for emission-time violation handling
- pre-lowering-spec-normalization.ts for upstream normalization
- catalog activity definitions that mark required properties
- final validation that surfaces critical activity viability issues

So the likely gap is not a missing enforcement framework. The likely gap is:
- a bug in the existing required-property preservation path
- and a lack of one consolidated artifact showing exactly where the property is lost

This task must not build a parallel required-property system. It must investigate and instrument the existing live path, consolidate diagnostics, and expose the exact stage where each required property is lost before any broad fix is attempted.

Goals
1. Trace the existing active-path required-property preservation flow end to end.
2. For each failing required property, identify the exact stage where it is lost.
3. Emit one structured diagnostics artifact explaining the preservation outcome clearly.
4. Patch only the specific failing branch(es) once runtime evidence proves the root cause.

Done looks like
- The pipeline emits a structured required-property diagnostics artifact for each critical activity/property under investigation.
- For each tracked required property, the artifact shows:
  - whether it existed in the source spec
  - whether it existed after pre-lowering normalization
  - whether it existed after lowering
  - whether it existed after required-property enforcement
  - whether it existed after compliance/normalization
  - whether it existed in final archive-bound XAML
  - exact failure stage and reason if lost
- Required-property failures no longer survive as opaque late structural-invalidity symptoms. They have explicit root-cause diagnostics.
- If one or more concrete failing branches are identified in the current logic, patch those branches without introducing a parallel required-property framework.

Out of scope
- Rebuilding required-property enforcement from scratch
- Broad mail-family/source-of-truth redesign
- Auto-wiring framework redesign
- Dependency-family redesign
- Business-value invention
- UI-only reporting changes without root-cause traceability

Tasks

1. Instrument existing required-property preservation path
Trace the existing live path for required properties through:
- spec generation output
- pre-lowering normalization
- critical-activity lowering
- required-property enforcement
- compliance/normalization
- final archive-bound XAML

For each tracked activity/property, record:
- workflow/file
- activityType
- requiredProperty
- presentInSpec
- presentAfterPreLoweringNormalization
- presentAfterLowering
- presentAfterRequiredPropertyEnforcement
- presentAfterCompliance
- presentInFinalXaml

2. Focus on benchmark cases first
Use at least these benchmark failures as the initial evidence targets:
- BirthdayGreetingsV26: GmailSendMessage.Body
- BirthdayGreetingsV26: UpdateEntity.EntityObject

Do not broaden to every property family until the diagnostics prove the actual failing stage for these benchmark cases.

3. Instrument existing enforcement and emission decision points
At the current live path, record for each tracked property:
- whether the enforcer saw the property
- whether a value was preserved
- whether a value was derived
- whether a sentinel or placeholder path was triggered
- whether emission-gate accepted, corrected, blocked, or downgraded the activity
- whether compliance/normalization rewrote or stripped the property shape

4. Emit consolidated diagnostics artifact
Add:
criticalActivityContractDiagnostics

Per activity/property include:
- workflow/file
- activityType
- requiredProperty
- presentInSpec
- presentAfterPreLoweringNormalization
- presentAfterLowering
- presentAfterRequiredPropertyEnforcement
- presentAfterCompliance
- presentInFinalXaml
- derivationApplied
- derivationSource
- sentinelPathTriggered
- emissionDecision
- lossStage
- rootCauseCategory
- remediationHint

Summary counts:
- totalCriticalActivitiesChecked
- totalRequiredPropertiesSatisfied
- totalRequiredPropertiesPreserved
- totalRequiredPropertiesDerived
- totalRequiredPropertiesLost
- totalEarlyContractFailures

5. Reuse existing diagnostics rather than creating competing truth
If existing diagnostics already capture part of this path, extend or consolidate them. Do not create a second conflicting required-property truth source.

6. Evidence-gated patching
Do not patch speculative root causes.
Any patch to required-property-enforcer, lowering, compliance, emission-gate, or emission logic must be justified by runtime evidence from:
- actual benchmark outputs
- or reproducible local pipeline runs

If evidence is not yet sufficient, stop after instrumentation/diagnostics and report the observed root-cause candidates.

7. Add focused regression tests
Cover:
- GmailSendMessage.Body present upstream and preserved through all stages
- UpdateEntity.EntityObject present upstream and preserved through all stages
- diagnostics clearly show the exact loss stage when a property disappears
- no parallel required-property framework is introduced
