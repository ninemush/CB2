FIX 1 — Service probe must emit progress 
events during its 60-second run

In chat/routes.ts, the service probe runs 
at line ~1019 and takes ~60 seconds. During 
this time the user sees "Getting things 
ready..." with no updates. The liveStatus 
event at line 1019 fires once but then 
nothing until the probe completes.

Add a keep-alive interval during the service 
probe that emits rotating liveStatus messages 
every 8 seconds:

  const probeMessages = [
    "Checking UiPath service availability...",
    "Connecting to Orchestrator...",
    "Verifying AI Center status...",
    "Almost ready to start generation..."
  ]
  let probeIdx = 0
  const probeKeepAlive = setInterval(() => {
    sendEvent(res, { 
      liveStatus: probeMessages[
        probeIdx % probeMessages.length
      ] 
    })
    probeIdx++
  }, 8000)

Clear this interval immediately after the 
service probe completes — before the 
liveStatus: "Starting UiPath package 
generation..." event fires at line 1193.

This covers the 60-second probe gap with 
meaningful messages.

FIX 2 — Doc-mode indicator must show real 
pipeline events not fabricated client steps

When the component switches to mode="doc" 
for docType="UiPath", the StreamingProgress 
Indicator currently shows fabricated steps 
from DOC_PROGRESS_STEPS["UiPath"] as a 
fallback. These are client-side guesses 
that do not reflect what the server is 
actually doing.

The PipelineLogPanel already exists and 
shows real server events correctly when 
pipelineLogEntries has content. The issue 
is that during the long tree enricher LLM 
call (2+ minutes), no pipelineEvent entries 
arrive so the log panel is empty and the 
fabricated steps show instead.

Two changes:

a) In workspace.tsx, when docType is 
   "UiPath" and pipelineLogEntries is 
   empty, show the liveStatus message 
   in the StreamingProgressIndicator 
   instead of cycling through fabricated 
   DOC_PROGRESS_STEPS. The liveStatus 
   will be set by the probe keep-alive 
   from Fix 1 and the generation progress 
   messages, so the user sees real server 
   messages instead of fabricated ones.

   Specifically: in the rendering condition 
   that decides between StreamingProgress 
   Indicator and PipelineLogPanel, when 
   docType === "UiPath" and 
   pipelineLogEntries.length === 0, 
   render StreamingProgressIndicator in 
   thinking mode using liveStatus as the 
   message rather than doc mode with 
   fabricated steps.

b) In document-routes.ts, add a keep-alive 
   inside the tree enricher wait that emits 
   pipelineEvent entries every 10 seconds 
   while the LLM call is in progress:

   { 
     stage: "ai_enrichment",
     status: "heartbeat", 
     summary: "AI is building the workflow 
       tree structure — this may take a 
       minute for complex processes...",
     timestamp: Date.now()
   }

   This gives the PipelineLogPanel something 
   real to show during the long LLM call 
   rather than being empty.
