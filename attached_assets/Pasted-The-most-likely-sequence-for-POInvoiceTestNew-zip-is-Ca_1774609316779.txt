Why The Output Looked Better Than It Was
The package has two truths:

the intended design truth: “we planned 7 workflows”
the artifact truth: “the ZIP contains 4 XAML files, 3 of them are stubs”
Right now CannonBall reports too much from the first truth and not enough from the second. The guide is effectively describing the pre-pruned, pre-finalized package model rather than the shipped artifact.
There’s also a second design problem: there are two DHG generation paths.
The legacy one is in xaml-generator.ts

The structured one is in dhg-generator.ts
That split makes consistency harder. The package assembler writes the legacy-style DHG, while the pipeline can also build a structured DHG in uipath-pipeline.ts. It’s a recipe for drift.
What To Change In Code
The minimum fixes I’d make are:
1. Build DHG inventory from final packaged files, not planned workflow names.
In package-assembler.ts, you already derive xamlContentsForDhg from deferredWrites. Do the same for names:
• collect final .xaml basenames from deferredWrites
• pass those names to the DHG instead of the earlier workflowNames
That alone would stop phantom files from appearing in the guide.
2. Recompute workflow inventory after structural dedup.
The current sequence decides workflowNames early in package-assembler.ts, then mutates the archive later. Any reporting that happens after pruning should derive from the post-prune archive state.
3. Do not let a stubbed Main.xaml drive reachability pruning the same way as a real Main.xaml.
If Main.xaml is replaced with a workflow-level stub, skip child-workflow pruning or preserve all originally generated workflows for developer handoff. Otherwise the system destroys evidence of what it intended to build.
4. Make readiness scoring artifact-aware.
If Main.xaml is workflow-stubbed, or if dispatcher/performer are stub-only, cap readiness hard. The 84% score is misleading for this artifact. The scoring in xaml-generator.ts should include:
count of workflow-level stubs
count of missing planned workflows
whether entry point is stubbed
whether final package graph is runnable
5. Fail or clearly downgrade package status when entrypoint is stubbed.
Shipping a stubbed Main.xaml should likely be FALLBACK_READY at best, not something that reads like near-production output.
6. Consolidate to one DHG generator.
Pick the structured outcome-report path in dhg-generator.ts as the source of truth, and feed it final archive-derived facts. Keeping both generators invites contradictions.
