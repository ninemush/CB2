These are bugs discovered when opening a generated POInvoiceReconciliation project in UiPath Studio 25.10.7 (Windows, VB). Fix these patterns in your XAML generation logic to produce error-free projects.
🔴 ISSUE 1 —
ExcelApplicationScope.Body
missing
ActivityAction
wrapper
What's wrong: The Body property of ExcelApplicationScope requires an ActivityAction wrapper around any child activity. Cannonball is placing a bare <Sequence> directly inside <ui:ExcelApplicationScope.Body>, which causes a load exception.
Broken pattern:
xml
<ui:ExcelApplicationScope.Body>
  <Sequence DisplayName="...">
    ...
  </Sequence>
</ui:ExcelApplicationScope.Body>
Correct pattern:
xml
<ui:ExcelApplicationScope.Body>
  <ActivityAction x:TypeArguments="x:Object">
    <ActivityAction.Argument>
      <DelegateInArgument x:TypeArguments="x:Object" Name="workbookApp" />
    </ActivityAction.Argument>
    <Sequence DisplayName="...">
      ...
    </Sequence>
  </ActivityAction>
</ui:ExcelApplicationScope.Body>
🔴 ISSUE 2 —
ContinueOnError
on activities that don't support it
What's wrong: Cannonball is adding ContinueOnError="True" or ContinueOnError="False" to activities that don't have this property. UiPath's XAML parser throws a hard load error when it encounters an unknown property on an activity type.
Activities that DO NOT support ContinueOnError:
ui:LogMessage
ui:Assign(in UiPath.System.Activities 26.x)
ui:AddQueueItem
ui:GetTransactionItem
While (System.Activities.Statements)
Rule: Only add ContinueOnError to activities that explicitly support it (e.g., ui:HttpClient, InvokeWorkflowFile, ui:Click, ui:TypeInto). When in doubt, omit it.
🔴 ISSUE 3 — Self-closing
<While>
with no Body
What's wrong: A <While> activity generated as self-closing (<While ... />) has no Body, which causes a load failure because WF requires a non-null Body.
Broken pattern:
xml
<While Condition="[hasMoreItems]" ContinueOnError="False" />
Correct pattern:
xml
<While Condition="[hasMoreItems]">
  <While.Body>
    <Sequence DisplayName="While Body">
      <!-- activities here -->
    </Sequence>
  </While.Body>
</While>
🔴 ISSUE 4 —
RetryScope
without namespace prefix
What's wrong: Cannonball generates <RetryScope .../> without the ui: namespace prefix and with no body. RetryScope lives in the UiPath namespace (ui:), not System.Activities. Without the prefix, XAML can't resolve the type → load error.
Broken pattern:
xml
<RetryScope NumberOfRetries="3" RetryInterval="00:00:05" ContinueOnError="False" />
Correct pattern:
xml
<ui:RetryScope NumberOfRetries="3" RetryInterval="00:00:05">
  <ui:RetryScope.Body>
    <Sequence>
      <!-- action to retry -->
    </Sequence>
  </ui:RetryScope.Body>
  <ui:RetryScope.Condition>
    <Sequence>
      <!-- condition check -->
    </Sequence>
  </ui:RetryScope.Condition>
</ui:RetryScope>
🔴 ISSUE 5 —
HttpClient
invalid attribute values
What's wrong: Three separate problems with ui:HttpClient generation:
5a —
Headers
uses JSON syntax instead of VB Dictionary
Broken:
xml
Headers="{ &quot;Authorization&quot;: &quot;Bearer &quot; + coupaApiToken }"
Correct (VB Dictionary expression in brackets):
xml
Headers="[New Dictionary(Of String, String)() From {{&quot;Authorization&quot;, &quot;Bearer &quot; &amp; coupaApiToken}, {&quot;Accept&quot;, &quot;application/json&quot;}}]"
5b —
URL
without expression brackets
Broken:
xml
URL="coupaBaseURL + &quot;/api/orders&quot;"
Correct:
xml
URL="[coupaBaseURL &amp; &quot;/api/orders&quot;]"
Note: VB uses & for string concatenation, not +. And all VB expressions in XAML attributes must be wrapped in [...].
5c —
Output
with descriptive text instead of variable name
Broken:
xml
Output="coupaApiToken (parsed from JSON response .access_token)"
Correct:
xml
Output="[coupaApiToken]"
5d —
ResponseType="Bytes"
is not a valid attribute
Remove it entirely. There is no ResponseType attribute on ui:HttpClient in UiPath.Web.Activities 1.18.x.
🔴 ISSUE 6 — Document Understanding activities without the DU package
What's wrong: Cannonball generates ui:DigitizeDocument, ui:ClassifyDocument, ui:ExtractDocumentData, ui:ValidateDocumentData — but these come from the UiPath.DocumentUnderstanding.Activities package which is not included in project.json. XAML can't resolve the types → load error.
Fix options (pick one):
Add the DU package to project.json dependencies when generating DU activities, OR
Don't generate DU activities unless the user explicitly requests them and the package is added
🔴 ISSUE 7 — Package version conflicts (Windows target framework)
What's wrong: The generated project.json included both:
UiPath.Excel.Activities 3.4.1
UiPath.UIAutomation.Activities 23.10.0
Both of these packages ship Microsoft.Office.Interop.Outlook.dll for the Windows target framework, causing a duplicate key crash during Studio's assembly resolution (GetAvailableAssemblies()).
Fix: When targeting the Windows framework, pin UiPath.UIAutomation.Activities to 25.10.x or later (where the Outlook DLL was removed). Avoid combining old UIAutomation versions with newer Excel.Activities.
Recommended safe package set for Windows / VB projects:
json
"UiPath.Excel.Activities": "3.4.1",
"UiPath.System.Activities": "26.2.1",
"UiPath.UIAutomation.Activities": "25.10.28",
"UiPath.Web.Activities": "[1.18.0]"
🟡 ISSUE 8 — Variables used in expressions but never declared
What's wrong: Many activities reference VB variables (e.g., coupaBaseURL, coupaApiToken, hasMoreItems) in their expression attributes, but those variables are never declared in <Sequence.Variables>. This causes validation errors in Studio (red exclamation marks).
Fix: For every variable referenced in any expression, ensure a corresponding <Variable> declaration exists in the nearest enclosing scope's <Sequence.Variables> block.
Variable declaration patterns:
xml
<Variable x:TypeArguments="x:String" Name="myStringVar" />
<Variable x:TypeArguments="x:String" Name="myStringWithDefault" Default="&quot;hello&quot;" />
<Variable x:TypeArguments="x:Boolean" Name="myBoolVar" Default="[True]" />
<Variable x:TypeArguments="s:Int32" Name="myIntVar" Default="[0]" />
<Variable x:TypeArguments="s:DateTime" Name="myDateVar" />
<Variable x:TypeArguments="x:Object" Name="myObjectVar" />
