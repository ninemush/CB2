This task restructures the UiPath XAML generation prompt from 
free-form generation to template-driven assembly. This is the 
highest-leverage quality improvement available — it prevents 
the entire class of schema errors at source rather than 
correcting them downstream.

The principle: the LLM's job is to ASSEMBLE and fill in VALUES, 
not to invent XAML syntax. Every structural pattern and every 
activity template is provided by the system. The LLM fills in 
placeholders only. This approach works correctly regardless of 
which LLM is used — the quality of the output depends on the 
templates and constraints, not on the LLM's training data 
knowledge of UiPath's XAML dialect.

PART 1 — Build the template library from the catalog
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Create server/catalog/xaml-template-builder.ts

This module reads the activity catalog and produces exact XML 
templates for every activity. These templates are injected into 
the LLM prompt at generation time. The templates are the single 
source of truth for correct XAML syntax — derived from the 
catalog, not from LLM training data.

TEMPLATE FORMAT

For each activity in the catalog, produce a template string 
showing the exact XML pattern with typed placeholders. 
Placeholders use {{name:type}} notation where type constrains 
what values are valid.

Placeholder types:
  {{name:string}}        — any string value
  {{name:expression}}    — a VB expression, wrapped in []
  {{name:varname}}       — a variable name, wrapped in []
  {{name:enum(A|B|C)}}   — must be exactly one of the listed values
  {{name:uniqueid}}      — generate as ActivityType_N (sequential)
  {{name:timespan}}      — a TimeSpan value e.g. 00:00:30
  {{name:bool}}          — True or False only

PART 2 — Rebuild the LLM generation prompt
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

In server/ai-xaml-enricher.ts (or wherever the XAML 
generation LLM call is constructed), replace the current 
prompt with a four-section structure.

This prompt architecture is LLM-agnostic. It does not rely 
on any model-specific capabilities, response formats, or 
training data knowledge. Any capable LLM given these 
instructions and templates will produce consistent output 
because the correct answer is fully specified — the LLM 
is filling in blanks, not making architectural decisions.

SECTION 1 — ROLE (fixed, never changes)

"You are a UiPath XAML assembler. Your sole responsibility 
is to produce valid UiPath Studio XAML by filling in the 
provided templates with values from the workflow 
specification.

ABSOLUTE RULES — these apply regardless of what the workflow 
specification requests:

1. You may ONLY use the XML patterns provided in the ACTIVITY 
   TEMPLATES section. Do not produce any XML element, 
   attribute, or structure not present in those templates. 
   If a step cannot be implemented with the provided 
   templates, output a ui:Comment explaining what is needed 
   and mark the enclosing Sequence DisplayName with the 
   prefix MANUAL:.

2. Fill in {{placeholder}} values only. The surrounding 
   XML structure is fixed and must not be modified.

3. {{name:enum(A|B|C)}} — output exactly one of the listed 
   values, nothing else.

4. {{name:expression}} — output the value wrapped in square 
   brackets: [your expression here]. String literals must 
   use: [&quot;your string here&quot;]

5. {{name:varname}} — output the variable name wrapped in 
   square brackets: [variableName]

6. {{name:uniqueid}} — generate a unique ID in format 
   ActivityType_N where N is a sequential integer starting 
   at 3. Increment N for each new element. Never reuse an ID.

7. {{name:timespan}} — output a TimeSpan string in format 
   HH:MM:SS e.g. 00:00:30 for 30 seconds.

8. {{name:bool}} — output exactly True or False, nothing else.

9. Every variable name referenced inside any OutArgument, 
   InArgument, or Condition must be declared in 
   Sequence.Variables before the first activity that uses 
   it. Use the variable declaration template. No exceptions.

10. Always wrap activities that can fail in the Standard 
    TryCatch template. Do not create custom error handling 
    structures.

11. The workflow header is fixed. Output it exactly as 
    provided. Do not alter any namespace declaration, 
    assembly reference, or root element attribute.

12. Output only the complete XAML document. No explanatory 
    text, no markdown, no code fences, no comments outside 
    of ui:Comment elements."

SECTION 3 — VARIABLE DECLARATION RULES (fixed)

"VARIABLE DECLARATION RULES

reference in any OutArgument, InArgument, or Condition.

Type mapping — use these exact strings, nothing else:
  Text values         → x:String
  True/False values   → x:Boolean

  Date and time       → s:DateTime
  Passwords           → s:SecureString
  Data tables         → s:DataTable
  Anything else       → x:Object

Variable naming convention to follow when creating new 
variable names:
  str_    for text (x:String)

  dt_     for data tables (s:DataTable)

PART 3 — Update the SDD decomposition stage
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The SDD decomposition stage currently produces free-form 
prose descriptions of what each workflow should do. This 
feeds into the generation prompt as unstructured text that 
the LLM must interpret.

Change it to produce structured step specs that reference 
template names directly. The generation LLM then has no 
interpretation to do — it looks up the named template and 
fills in the specified values.

Update the decomposition LLM call with this instruction 
added to its prompt:

"For each step in the workflow, output a structured spec 
in this exact format:

STEP [N]:
  template: [exact template name from the list below]
  displayName: [human-readable name for this activity]
  errorHandling: [yes/no — wrap in standard TryCatch]
  properties:
    [propertyName]: [value or expression]
    [propertyName]: [value or expression]
  outputVar: [variable name for output, if any]
  outputType: [x:String / x:Boolean / x:Int32 etc]

If a step cannot be implemented with the available 
templates, output:
  template: MANUAL
  displayName: [what needs to be implemented]
  reason: [why no template covers this]

Available template names:
[inject template name list from catalog]

Rules:
- template must be an exact match to a name in the list 
  above or MANUAL
- Every property value must be a concrete value or VB 
  expression — no vague descriptions
- outputVar must follow the naming convention: 
  str_ for strings, b_ for booleans, int_ for integers, 
  sec_ for passwords, no prefix for objects
- errorHandling: yes for any step that calls an external 
  system, reads credentials, or can fail at runtime"

This produces a machine-readable spec. The generation 
stage parses it and constructs the prompt Section 4 from 
it. No free-form interpretation at generation time.

stricter because the expected output shape is now fully

1. For each activity element in generated XAML, verify 
   it matches its template:
   - All required attributes present

   Surface this in the CannonBall UI as a generation 
   quality indicator so it is visible without digging 
   into logs.
