We have a systemic transport/state-contract problem in the UiPath generation flow.

Observed behavior:
- The frontend starts UiPath generation correctly.
- The backend completes generation successfully.
- But the UI marks the build as FAILED because it does not receive streamed progress events within the safety timeout window.
- Later, the full response arrives at once, indicating proxy buffering rather than backend failure.

Important conclusion:
This is not primarily a package-builder failure.
It is a streaming/proxy-buffering + frontend state contract failure.
CannonBall is incorrectly treating “no progress events received yet” as “build failed.”

What I want:
Design and implement an upstream fix that makes UiPath generation resilient even when SSE progress is buffered by a reverse proxy.

Required goals:
1. Do not mark a build as FAILED just because progress events were not received within the timeout window while the request is still pending.
2. Distinguish:
   - build execution state
   - stream health / progress visibility state
3. Make SSE responses as anti-buffering as possible.
4. Ensure the UI can still converge to the correct final result even if progress events arrive late or all at once.
5. Preserve CannonBall’s UX trustworthiness: false failure states must be eliminated.

Implementation direction:
1. Harden SSE responses in generate-uipath and similar long-running endpoints:
   - set anti-buffering headers such as:
     - Cache-Control: no-cache, no-transform
     - X-Accel-Buffering: no
     - Connection: keep-alive
   - flush headers immediately
   - send an immediate early progress event before expensive work begins
   - keep heartbeat/progress events flowing frequently

2. Change the frontend timeout contract:
   - the timeout should not set status to FAILED merely because no progress event was seen
   - if fetch is still pending / stream still open, transition to a non-terminal state such as:
     - waiting for server updates
     - connection delayed
     - still processing
   - FAILED should only be set when:
     - server explicitly emits failed status
     - fetch aborts/errors
     - request is definitively terminated without success

3. Add a recovery path for buffered responses:
   - when the buffered response finally arrives, the UI must reconcile to the true final result
   - no stale FAILED badge should survive if the server actually completed successfully

4. If needed, propose a follow-up architecture:
   - server-side generation run tracking
   - client polling fallback when SSE visibility is unreliable

Please deliver:
1. concise root-cause summary
2. exact frontend/backend changes needed now
3. whether a polling/run-status fallback is needed immediately or as phase 2
4. implementation plan in priority order

Important:
- Do not treat this as a timer-only fix.
- Do not just increase the timeout again.
- Solve the false-failure contract at the root.
