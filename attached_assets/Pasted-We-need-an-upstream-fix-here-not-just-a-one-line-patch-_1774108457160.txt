We need an upstream fix here, not just a one-line patch.

Problem:
SDD generation is failing because persisted chat history includes messages with role = "system", and the document-generation path is passing them into the Anthropic Messages API as normal input messages. Anthropic requires system instructions to be provided via the top-level system parameter, not as message entries.

Replit’s diagnosis is directionally correct:
- trimChatForDocGen is allowing system-role messages through
- TypeScript casts are hiding the problem at compile time
- Anthropic rejects the runtime payload

But I want this fixed as a chat-to-LLM message contract issue, not just a local filter.

What I want:
Create an upstream fix that guarantees any Anthropic messages array is valid by construction.

Required fixes:
1. Add a reusable message-normalization/sanitization helper for document generation
   - only allow user and assistant roles into the final messages array
   - explicitly exclude or separately handle system-role entries
   - do not rely on unsafe role casts

2. Fix trimChatForDocGen (or the equivalent document-generation preparation path)
   - filter out system-role messages
   - preserve the existing filtering for internal markers like [DOC:] and [UIPATH:]
   - ensure the returned message array is runtime-safe for Anthropic

3. Add runtime validation before Anthropic document-generation calls
   - if any message role other than user/assistant survives, fail fast with a clear internal error before calling the API
   - do not let invalid roles reach the provider

4. Remove unsafe role coercion in this path
   - avoid patterns like:
     - m.role as "user" | "assistant"
   - prefer explicit normalization and validation

5. Check whether any other doc-generation or chat-summary paths reuse the same flawed assumption
   - if so, route them through the same normalization helper

Deliverable:
1. concise root-cause summary
2. upstream implementation task with:
   - objective
   - done looks like
   - out of scope
   - tasks
   - relevant files
3. acceptance criteria that require:
   - no system-role messages ever appear in Anthropic messages arrays
   - SDD generation succeeds past this point
   - invalid chat roles are caught internally before the provider call
   - no unsafe role casts remain in the affected path

Important:
- This is an upstream LLM message-contract fix
- Do not solve it with a one-off local patch only
- The system should produce Anthropic-valid message arrays by construction
